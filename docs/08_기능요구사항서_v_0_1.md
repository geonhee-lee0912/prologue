이 문서는 프롤로그 MVP의 주요 기능을 개발 가능한 수준으로 정의한다.

각 기능은 사용자 관점의 요구사항, 기본 규칙, 예외 조건, 연결 화면, 연결 데이터를 포함한다.

이 문서는 다음 작업의 기준 문서로 사용된다.

1. 개발 태스크 분해
2. API 설계
3. 데이터베이스 설계
4. QA 시나리오 작성
5. MVP 범위 조정
6. 운영 정책 검토

# 4. 계정/가입

## FR-A01. 회원가입

| 항목 | 내용 |
|---|---|
| 기능 ID | FR-A01 |
| 기능명 | 회원가입 |
| 사용자 | 신규 사용자 |
| 우선순위 | P1 |
| 설명 | 사용자가 본인 인증을 거쳐 프롤로그 계정을 생성한다. |
| 관련 화면 | A03, A04 |
| 관련 데이터 | User, UserAuth, UserConsent |

### 요구사항

1. 회원가입은 본인 인증 (FR-B01) 을 1회 거쳐 완료한다.
   - 휴대폰 본인 인증 진입 또는 카카오 OAuth + 본인 인증 진입 중 선택
2. 본인 인증 결과로 휴대폰·CI·생년월일·성별을 한 번에 확보한다.
3. 가입 시 필수 약관과 선택 약관을 구분해 동의받아야 한다.
4. 필수 약관에 동의하지 않으면 가입을 진행할 수 없다.
5. 가입 완료 시점에 `User.identity_verified=true` 로 저장한다 (FR-B01 자동 충족).
6. 가입 완료 후 얼굴 인증 (FR-B02) 단계로 이동한다.
7. 동일 CI로 중복 가입할 수 없다 (`UserAuth.identity_ci_hash` UNIQUE).
8. 가입 시 기본 계정 상태는 `pending` 으로 생성한다.
9. 필수 인증과 필수 프로필 작성 완료 후 계정 상태를 `active` 로 전환한다.

### 예외/정책

- 탈퇴 후 재가입 제한 기간이 있다면 별도 정책으로 정의한다.
- 만 19세 미만 또는 서비스 대상 연령 밖 사용자는 가입 제한 또는 대기 처리한다.
- 휴대폰 OTP 는 회원가입이 아닌 **로그인 (FR-A02) 용도** 로만 사용한다.

## FR-A02. 로그인

| 항목 | 내용 |
|---|---|
| 기능 ID | FR-A02 |
| 기능명 | 로그인 |
| 사용자 | 기존 사용자 |
| 우선순위 | P1 |
| 설명 | 기존 사용자가 자신의 계정으로 서비스에 접근한다. |
| 관련 화면 | A03 |
| 관련 데이터 | User, RefreshToken |

### 요구사항

1. 로그인 방식: 휴대폰 OTP 또는 카카오 OAuth. **본인 인증 (PASS) 재호출 없음.**
2. 휴대폰 OTP 로그인: 가입 시 등록한 휴대폰 번호로 1회용 코드를 받아 검증한다.
3. 카카오 로그인: 카카오 OAuth access token 으로 기존 User 를 조회·로그인한다.
4. 로그인 성공 시 access token 과 refresh token 을 반환한다 (POLICY: access 15분 / refresh 30일).
5. 정지된 계정은 로그인 후 서비스 이용 제한 안내를 받는다.
6. 탈퇴한 계정은 로그인할 수 없다.
7. 본인 인증 미완료 사용자는 로그인 후 인증 단계로 이동한다 (이상 상태 — 일반 흐름 아님).
8. 프로필 미완성 사용자는 로그인 후 프로필 작성 단계로 이동한다.

### 예외/정책

- 동일 IP 또는 동일 번호의 OTP 요청은 분당 제한 (Throttler 적용).
- OTP 는 3분 TTL, 최대 5회 시도.

# 5. 인증

## FR-B01. 본인 인증

| 항목 | 내용 |
|---|---|
| 기능 ID | FR-B01 |
| 기능명 | 본인 인증 |
| 사용자 | 가입 사용자 |
| 우선순위 | P1 |
| 설명 | 가짜 계정과 중복 계정을 줄이기 위해 PASS/NICE 본인 인증을 수행한다. |
| 관련 화면 | A03 (가입 흐름 내부에서 진행) |
| 관련 데이터 | UserAuth |

### 요구사항

1. **본인 인증은 회원가입 (FR-A01) 과 동일 트랜잭션에서 수행한다** — 별도 단계 아님.
2. 인증 결과는 `UserAuth.identity_verified=true`, `identity_verified_at`, `identity_provider` 에 저장한다.
3. 인증 응답의 CI 만 `UserAuth.identity_ci_hash` 로 해시 저장 (`@unique`). 이름·생년월일은 검증 후 폐기한다.
4. 본인 인증 실패 시 가입 화면에서 재시도할 수 있어야 한다.
5. 인증 결과는 프로필 배지로 표시할 수 있다.
6. **본인 인증 미완료 사용자는 추천 후보가 될 수 없다** (가입 흐름 중단 시).
7. 동일 CI 로 중복 가입할 수 없다 (UNIQUE 제약).

### 예외/정책

- 인증 실패 횟수가 과도한 경우 일정 시간 재시도를 제한할 수 있다.
- 인증 업체 연동 실패 시 재시도 또는 고객센터 문의를 안내한다.
- 카카오 OAuth 로 진입한 사용자도 본인 인증 1회를 거쳐야 한다 (카카오는 CI 미제공).
- MVP 단계: `MockIdentityVerificationProvider` 사용. PASS/NICE 계약 후 provider 만 교체.

## FR-B02. 얼굴 인증

| 항목 | 내용 |
|---|---|
| 기능 ID | FR-B02 |
| 기능명 | 얼굴 인증 |
| 사용자 | 가입 사용자 |
| 우선순위 | P1 |
| 설명 | 사진 도용과 허위 프로필 가능성을 줄인다. |
| 관련 화면 | B03 |
| 관련 데이터 | UserAuth, Photo, AdminReview |

### 요구사항

1. 사용자는 대표 사진과 얼굴 인증 절차를 진행해야 한다.
2. 얼굴 인증이 완료되어야 추천 수신 상태가 활성화된다.
3. 얼굴 인증 실패 시 재촬영 또는 재업로드를 안내한다.
4. 얼굴 인증 결과는 `not_submitted`, `pending`, `verified`, `rejected` 상태로 저장한다.
5. 자동 인증이 어려운 경우 운영자 검수 상태인 `pending`으로 둘 수 있다.
6. 얼굴 인증이 반려된 경우 반려 사유를 안내한다.

## FR-B04. 관계 목적 설문

| 항목 | 내용 |
|---|---|
| 기능 ID | FR-B04 |
| 기능명 | 관계 목적 설문 |
| 사용자 | 가입 사용자 |

| 설명 | 사용자의 만남 목적과 관계 속도를 추천 기준으로 수집한다. |
| 관련 화면 | B05 |
| 관련 데이터 | RelationshipPreference |

### 요구사항

1. 사용자는 추천을 받기 전에 관계 목적 설문을 완료해야 한다.

4. 사용자는 마이페이지에서 설문 결과를 수정할 수 있어야 한다.
5. 수정된 설문 결과는 이후 추천부터 반영한다.
6. 설문 완료 시 `relationship_survey_completed=true`로 저장한다.

## FR-C01. 사진 등록

| 항목 | 내용 |
|---|---|
| 기능 ID | FR-C01 |
| 기능명 | 사진 등록 |
| 사용자 | 가입 사용자 |
| 우선순위 | P1 |
| 설명 | 사용자가 프로필에 표시할 사진을 등록한다. |
| 관련 화면 | C02 |
| 관련 데이터 | Photo |

### 요구사항

1. 사용자는 대표 사진을 1장 이상 등록해야 한다.
2. 대표 사진이 없으면 추천 대상이 될 수 없다.
3. 사진은 운영 정책에 따라 검수 상태를 가질 수 있다.
4. 부적절한 사진은 반려 처리할 수 있어야 한다.
5. 사용자는 사진을 추가, 삭제, 대표 사진 변경할 수 있어야 한다.
6. 얼굴 인증과 대표 사진 간 검증이 필요할 수 있다.

## FR-C02. 기본 정보 입력

| 항목 | 내용 |
|---|---|
| 기능 ID | FR-C02 |
| 기능명 | 기본 정보 입력 |

| 관련 화면 | C03 |
| 관련 데이터 | User, Profile |

### 요구사항

1. 사용자는 나이, 성별, 거주 지역, 직업군을 입력해야 한다.
2. 나이는 본인 인증 결과와 일치해야 한다.

5. 사진, 동영상, 음성, 파일, 위치, 자유 링크 전송은 MVP에서 지원하지 않는다.
6. 대화방에는 신고, 차단, 연락처 교환 요청 기능이 있어야 한다.

| 추천 후보 생성 | 추천 카드 | 추천 결과 표시 |
| 추천 카드 | 관심/넘기기 | 사용자 반응 수집 |
| 관심 보내기 | 상호 매칭 | 양방향 관심 시 매칭 |

# 15. 주요 정책 메모

1. 무료 사용자와 Plus 사용자는 기본적으로 서로 매칭될 수 있다.
2. 유료 차이는 매칭 가능 여부가 아니라 추천 수, 리포트 깊이, 대화 보조, 대화 연장, 추가 추천권에서 만든다.
3. 내부 채팅은 MVP에 포함하되 텍스트 중심으로 제한한다.
4. 연락처 교환은 반드시 상호 동의 기반으로 한다.
5. 거절 사유는 추천 개선 목적 외에 상대 평가나 공개용으로 사용하지 않는다.
6. 인증 원본은 최소 보관하고 인증 결과 중심으로 관리한다.
7. 신고/차단은 프로필 상세와 대화방에서 항상 접근 가능해야 한다.
