# 1. 문서 목적

이 문서는 프롤로그의 가입, 인증, 관계 목적 확인, 프로필 작성까지 이어지는 초기 온보딩 흐름을 정의하기 위한 문서다.

프롤로그의 온보딩은 단순 회원가입 절차가 아니다. 사용자가 서비스의 핵심 가치인 신뢰, 진정성, 큐레이션, 이야기형 프로필, 안전감을 처음 경험하는 구간이다. 온보딩이 너무 가볍다면 프롤로그의 신뢰 포지션이 약해지고, 너무 무겁다면 결정사처럼 느껴져 이탈이 커질 수 있다.

따라서 이 문서는 다음 목표를 가진다.

1. 필수 가입 단계와 선택 단계를 분리한다.
2. 인증 절차의 목적과 공개 범위를 명확히 설명한다.
3. 관계 목적 설문이 결혼 압박처럼 느껴지지 않도록 설계한다.
4. 이야기형 프로필 작성을 부담 없이 유도한다.
5. 사용자가 추천을 받기 위한 준비가 완료되었다고 느끼게 한다.
6. MVP에서 필요한 최소 온보딩 범위를 정의한다.

---

# 2. 온보딩 한 줄 정의

**프롤로그의 온보딩은 사용자가 신뢰 가능한 추천을 받기 위해 필요한 기본 인증, 관계 목적, 이야기형 프로필을 부담 없이 완성하도록 돕는 첫 경험이다.**

핵심 원칙은 다음과 같다.

> 필요한 확인은 분명하게, 입력 부담은 가볍게, 공개 범위는 투명하게.

---

# 3. 온보딩 설계 원칙

## 3.1 인증은 심사가 아니라 신뢰 장치로 설명한다

사용자가 인증을 “평가받는 과정”으로 느끼면 이탈이 커진다. 프롤로그는 인증을 허위 프로필, 도용 사진, 목적 불명확성을 줄이기 위한 최소 신뢰 장치로 설명한다.

## 3.2 전체 과정을 먼저 보여준다

사용자는 무엇을 얼마나 입력해야 하는지 모를 때 부담을 느낀다. 온보딩 시작 전 전체 단계를 간단히 보여주고 예상 소요 시간을 안내한다.

## 3.3 필수와 선택을 분리한다

초기 추천을 받기 위해 반드시 필요한 정보와, 추천 품질을 높이는 선택 정보를 분리한다. 직업/재직 인증, 학력 인증, 경제 안정성 인증 등은 단계적으로 유도한다.

## 3.4 결혼을 압박하지 않는다

프롤로그는 진지한 관계를 다루지만 결혼중개 서비스처럼 보이면 안 된다. 관계 목적 설문은 결혼 여부보다 관계의 속도, 연락 방식, 장기 가능성, 만남 태도 중심으로 설계한다.

## 3.5 프로필 작성은 짧은 문장부터 시작한다

이야기형 프로필은 프롤로그의 차별점이지만, 처음부터 긴 자기소개를 요구하면 부담이 크다. 선택형, 키워드형, 짧은 문장형을 조합한다.

## 3.6 공개 범위를 반복해서 안심시킨다

사용자는 인증 정보, 사진, 관계 목적 답변이 상대에게 어떻게 보이는지 궁금해한다. 주요 입력 단계마다 공개 범위와 사용 목적을 안내한다.

---

# 4. 온보딩 전체 흐름

MVP 기준 온보딩 흐름은 다음과 같다.

> 시작 안내 → 계정 생성 → 기본 정보 입력 → 본인 인증 → 얼굴 인증 → 싱글 상태/매너 서약 → 관계 목적 설문 → 프로필 작성 → 프로필 공개 범위 확인 → 추천 준비 완료

선택 인증은 온보딩 중간에 강제하지 않고, 추천 준비 완료 후 또는 프로필 완성도 화면에서 유도할 수 있다.

---

# 5. 단계별 상세 기획

## 5.1 시작 안내 화면

### 목적

사용자가 프롤로그의 성격과 온보딩 절차를 이해하게 한다.

### 핵심 메시지

- 프롤로그는 가볍지 않은 만남을 위한 인증 소개팅이다.
- 좋은 추천을 위해 몇 가지 확인과 프로필 작성이 필요하다.
- 입력한 정보는 공개 범위에 따라 관리된다.

### 화면 요소

- 서비스 한 줄 설명
- 온보딩 예상 단계
- 예상 소요 시간
- 시작하기 버튼
- 개인정보/공개 범위 안내 링크

### 예시 문구

> 신뢰할 수 있는 추천을 위해 몇 가지 확인이 필요해요.  
> 대부분의 정보는 배지나 요약 형태로만 표시되며, 공개 범위는 직접 설정할 수 있습니다.

---

## 5.2 계정 생성

### 목적

서비스 이용을 위한 기본 계정을 만든다.

### 수집 항목

- 휴대폰 번호 또는 소셜 로그인
- 약관 동의
- 개인정보 처리방침 동의
- 만 19세 이상 확인
- 마케팅 수신 동의: 선택

### 정책

- 관계 매칭 서비스 특성상 휴대폰 기반 본인 확인을 우선 검토한다.
- 소셜 로그인은 편의 기능으로 제공할 수 있으나 본인 인증을 대체하지 않는다.

### 이탈 리스크

- 약관이 많아 부담스러움
- 개인정보 수집 불안

### 대응

- 필수/선택 동의를 명확히 분리한다.
- 개인정보 사용 목적을 짧게 요약한다.

---

## 5.3 기본 정보 입력

### 목적

추천과 필터링에 필요한 최소 기본 정보를 입력한다.

### 필수 항목

- 이름 또는 서비스 표시명
- 생년월일
- 성별
- 거주 지역
- 선호 상대 성별
- 추천 가능 지역 또는 생활권

### 선택 항목

- 직업군
- 키
- 흡연 여부
- 음주 성향
- 종교
- 자녀 여부 등 민감할 수 있는 항목은 초기 MVP에서 신중히 검토

### 공개 방식

- 이름은 실명 전체 공개가 아니라 표시명 또는 일부 마스킹을 검토한다.
- 생년월일은 나이로만 표시한다.
- 거주지는 시/구 단위 등 범위 조절이 필요하다.

### 예시 문구

> 추천에 필요한 최소 정보만 먼저 확인할게요.  
> 자세한 공개 범위는 이후에 직접 조정할 수 있습니다.

---

## 5.4 본인 인증

### 목적

가짜 계정과 중복 계정을 줄인다.

### 인증 방식 후보

- 휴대폰 본인 인증
- PASS/통신사 인증
- 신분증 인증은 MVP에서는 부담이 크므로 우선순위 낮음

### 사용자에게 보여줄 설명

- 본인 인증은 허위 계정을 줄이기 위한 과정이다.
- 인증 결과는 상대에게 실명으로 공개되지 않는다.
- 인증 여부는 신뢰 배지로 표시될 수 있다.

### 완료 후 상태

- 본인 인증 완료 배지
- 중복 가입 여부 확인
- 나이 검증

### 예시 문구

> 본인 인증은 허위 계정을 줄이고, 더 신뢰할 수 있는 추천을 만들기 위한 과정입니다.

---

## 5.5 얼굴 인증

### 목적

도용 사진과 허위 프로필을 줄인다.

### 인증 방식 후보

- 셀피 촬영
- 프로필 사진과 셀피 비교
- 수동 검수 또는 외부 API 활용

### 사용자에게 보여줄 설명

- 얼굴 인증은 도용 사진을 줄이기 위한 과정이다.
- 인증용 이미지는 상대에게 공개되지 않는다.
- 프로필 사진과 인증 결과는 별도로 관리된다.

### 실패 케이스

- 얼굴 인식 실패
- 사진 화질 부족
- 프로필 사진과 불일치
- 반복 실패

### 대응

- 재촬영 가이드 제공
- 조명/각도 안내
- 반복 실패 시 검토 요청

### 예시 문구

> 얼굴 인증은 도용 사진을 줄이기 위한 과정이에요.  
> 인증용 사진은 상대에게 공개되지 않습니다.

---

## 5.6 싱글 상태 서약

### 목적

기혼자, 연인 있음, 관계 목적이 불성실한 사용자를 줄인다.

### 정책 방향

초기에는 증빙 인증보다 서약 중심으로 운영한다. 고신뢰 풀 또는 프리미엄 풀에서는 증빙 인증을 추가 검토할 수 있다.

### 서약 문구 예시

> 현재 배우자 또는 연인이 없으며, 새로운 관계를 시작할 의사가 있음을 확인합니다.

### 보조 문구

> 프롤로그는 신뢰할 수 있는 만남을 위해 싱글 상태 서약을 받고 있습니다. 허위 서약이 확인될 경우 서비스 이용이 제한될 수 있습니다.

### 주의점

- “혼인상태 인증”이라는 표현은 초기 전면에서 사용하지 않는다.
- 법적 미혼 여부만이 아니라 현재 관계 상태를 확인하는 방향으로 설명한다.

---

## 5.7 매너 서약

### 목적

잠수, 노쇼, 무례한 대화, 허위 정보 등 비매너 행동을 예방한다.

### 서약 항목 예시

- 허위 정보를 입력하지 않겠습니다.
- 상대를 존중하는 태도로 대화하겠습니다.
- 불쾌하거나 위협적인 표현을 사용하지 않겠습니다.
- 만남 약속을 가볍게 취소하거나 노쇼하지 않겠습니다.
- 기혼자, 연인 있음 등 관계 상태를 속이지 않겠습니다.

### 제품 역할

매너 서약은 법적 면책 수단이 아니라 서비스 문화 형성을 위한 장치다. 신고/제재 정책과 연결되어야 한다.

---

## 5.8 관계 목적 설문

### 목적

사용자가 원하는 관계의 온도, 속도, 장기 가능성을 확인한다.

### 질문 설계 원칙

- 결혼을 직접 압박하지 않는다.
- “정답”이 있는 질문처럼 느껴지지 않게 한다.
- 추천 품질 향상 목적을 설명한다.
- 선택지 간 위계가 생기지 않도록 표현한다.

### 질문 예시

#### Q1. 지금 어떤 만남을 원하시나요?

- 천천히 알아가는 만남
- 자연스럽게 시작하는 연애
- 진지한 연애
- 장기적인 관계를 염두에 둔 만남

#### Q2. 좋은 사람을 만난다면 어느 정도의 관계까지 생각해볼 수 있나요?

- 아직은 연애 중심
- 장기 연애까지 가능
- 결혼 가능성도 열려 있음
- 구체적인 미래를 함께 생각하고 싶음

#### Q3. 관계를 시작할 때 선호하는 속도는 어떤가요?

- 천천히 알아가는 편
- 자연스럽게 가까워지는 편
- 확신이 들면 빠르게 가까워지는 편

#### Q4. 연락 빈도는 어떤 편이 편한가요?

- 천천히, 부담 없이
- 하루에 한두 번은 꾸준히
- 자주 연락하는 편이 좋음

### 예시 안내 문구

> 관계를 시작하는 속도와 온도가 비슷한 사람을 추천하기 위한 질문이에요.

---

## 5.9 프로필 사진 등록

### 목적

상대가 사용자를 기본적으로 이해할 수 있는 사진 정보를 제공한다.

### 권장 구조

- 대표 사진 1장: 필수
- 추가 사진 2~4장: 권장
- 일상/취향 사진: 선택

### 사진 가이드

권장:

- 얼굴이 잘 보이는 사진
- 최근 사진
- 과도한 보정이 없는 사진
- 일상이나 취향이 드러나는 사진

비권장:

- 얼굴이 가려진 사진
- 단체 사진만 등록
- 과도한 노출
- 타인의 사진
- 유명인/캐릭터 이미지

### 공개 범위 고려

사진 공개 범위는 추후 개인정보/공개범위 정책서에서 확정한다. MVP에서는 추천 카드와 프로필 상세에서 기본 공개하되, 고도화 단계에서 단계적 공개를 검토한다.

---

## 5.10 이야기형 프로필 작성

### 목적

사용자가 조건표가 아니라 자신의 이야기와 관계 성향을 표현하게 한다.

### 작성 방식

- 짧은 문장형 질문
- 선택형 키워드
- 다중 선택형 라이프스타일
- 자유 입력은 최소화하되 핵심 문항에는 허용

### 필수 문항 예시

1. 나를 가장 잘 설명하는 한 문장
2. 요즘 가장 즐겁게 하는 일
3. 좋은 관계에서 중요하게 생각하는 것
4. 주말을 보내는 방식
5. 대화할 때 편안한 분위기

### 선택 문항 예시

- 최근 빠져 있는 관심사
- 나에게 중요한 생활 리듬
- 친해지는 데 필요한 시간
- 상대에게 고마움을 느끼는 순간

### 예시 문구

> 긴 글을 쓰지 않아도 괜찮아요.  
> 당신다운 짧은 문장이 좋은 추천의 시작이 됩니다.

---

## 5.11 프로필 공개 범위 확인

### 목적

사용자가 자신이 입력한 정보가 어떻게 보이는지 확인하고 안심하게 한다.

### 화면 요소

- 상대에게 보이는 내 프로필 미리보기
- 공개되는 정보 목록
- 공개되지 않는 정보 목록
- 인증 원본 미공개 안내
- 수정하기 버튼

### 예시 문구

> 상대에게는 인증 원본이 아닌 인증 완료 배지만 표시됩니다.  
> 공개 범위는 이후에도 변경할 수 있어요.

## 5.12 추천 준비 완료

### 목적

사용자가 온보딩을 완료했다는 성취감과 첫 추천에 대한 기대를 느끼게 한다.

### 화면 요소

- 추천 준비 완료 메시지
- 프로필 완성도
- 완료한 인증 배지
- 추가 인증 안내
- 첫 추천 예상 안내
- 홈으로 이동

### 예시 문구

> 추천 준비가 완료되었어요.  
> 프롤로그 큐레이터가 당신의 관계 성향과 프로필을 바탕으로 어울릴 만한 사람을 정리할게요.

# 7. 온보딩 분기 전략

## 7.1 기본형 온보딩

모든 사용자가 거치는 기본 흐름이다.

> 계정 생성 → 본인 인증 → 얼굴 인증 → 싱글/매너 서약 → 관계 목적 설문 → 기본 프로필 작성 → 추천 준비 완료

## 7.2 빠른 시작형 온보딩

이탈을 줄이기 위해 최소 입력 후 추천 준비 상태로 진입하는 방식이다. 단, 프롤로그의 신뢰 포지션을 해치지 않도록 필수 인증은 유지한다.

> 계정 생성 → 본인/얼굴 인증 → 관계 목적 핵심 3문항 → 최소 프로필 → 추천 준비 완료 → 이후 프로필 보강 유도

## 7.3 고신뢰형 온보딩

직업/재직 인증, 추가 문항, 더 높은 프로필 완성도를 요구하는 흐름이다. 초기 MVP에서는 전체 필수로 두지 않고 선택 또는 프리미엄 풀 후보로 둔다.

> 기본형 온보딩 완료 → 직업/재직 인증 → 추가 프로필 작성 → 고신뢰 추천 풀 안내

# 8. 온보딩 카피 원칙

## 8.1 인증 안내

권장:

> 더 신뢰할 수 있는 추천을 위해 본인 인증이 필요해요.

비권장:

> 심사를 통과해야 가입할 수 있습니다.

## 8.2 관계 목적 설문

권장:

> 관계를 시작하는 속도와 온도가 비슷한 사람을 추천하기 위한 질문이에요.

비권장:

> 결혼 의향을 정확히 선택해주세요.

## 8.3 프로필 작성

권장:

> 완벽한 답보다, 당신다운 답이 더 중요해요.

비권장:

> 매력적인 자기소개를 작성해야 높은 평가를 받을 수 있어요.

## 8.4 공개 범위 안내

권장:

> 인증 원본은 상대에게 공개되지 않고, 인증 완료 배지로만 표시됩니다.

비권장:

> 걱정하지 마세요. 안전하게 보호됩니다.

# 9. 이탈 리스크와 대응

| 단계 | 이탈 원인 | 대응 전략 |
|---|---|---|
| 시작 안내 | 절차가 길어 보임 | 단계 수와 예상 시간 명시 |
| 계정 생성 | 약관/개인정보 부담 | 필수/선택 동의 분리 |
| 본인 인증 | 개인정보 불안 | 공개 범위와 사용 목적 안내 |
| 얼굴 인증 | 사진 공개 우려 | 인증용 사진 미공개 안내 |
| 관계 목적 설문 | 결혼 압박처럼 느껴짐 | 관계 속도/온도 중심 질문 |
| 프로필 작성 | 글쓰기 부담 | 예시, 선택형, 짧은 문항 제공 |
| 추가 인증 | 과도한 요구 | 선택 인증으로 분리, 보상 설명 |
| 추천 대기 | 기대 하락 | 추천 준비 완료와 예상 안내 |
