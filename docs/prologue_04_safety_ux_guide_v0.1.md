# 1. 문서 목적

이 문서는 프롤로그 사용자가 서비스 전반에서 안전감과 통제감을 느끼도록 하기 위한 UX 원칙, 화면 구성, 문구, 인터랙션 기준을 정의한다.

프롤로그는 인증 기반 진지한 소개팅 앱이다. 사용자는 프롤로그에 가입할 때 상대가 실제 사람인지, 관계 목적이 진지한지, 내 사진과 정보가 안전하게 다뤄지는지, 불편한 상대를 만났을 때 서비스가 제대로 대응하는지를 중요하게 본다.

따라서 안전 UX는 단순히 신고 버튼을 넣는 문제가 아니다. 사용자가 온보딩, 인증, 프로필 작성, 추천 카드, 매칭 리포트, 대화, 차단/신고, 탈퇴에 이르기까지 “내가 통제할 수 있다”, “서비스가 기준을 가지고 있다”, “내 정보가 함부로 공개되지 않는다”고 느끼게 해야 한다.

이 문서는 다음 의사결정의 기준으로 활용한다.

1. 사용자에게 안전감을 주는 화면과 문구 원칙을 정한다.
2. 인증과 개인정보 안내를 불안하게 만들지 않고 이해시키는 방식을 정한다.
3. 신고/차단 기능의 위치와 문구를 정한다.
4. 대화 중 비매너와 위험 행동을 줄이는 UX 장치를 정의한다.
5. 여성 사용자의 안전감과 통제감을 강화하는 제품 기준을 만든다.
6. 안전 관련 화면에서 책방/서점 메타포 사용 강도를 조절한다.

# 2. 안전 UX 한 줄 정의

**프롤로그의 안전 UX는 사용자가 만남을 시작하기 전부터 대화를 끝낼 때까지, 자신의 정보와 관계 진행을 직접 통제하고 있다고 느끼게 하는 경험 설계다.**

사용자에게 전달할 핵심 메시지는 다음과 같다.

> 프롤로그는 신중한 만남을 위해, 확인할 것은 확인하고 공개할 것은 사용자가 정하게 합니다.

# 3. 안전 UX의 핵심 원칙

## 3.1 안전하다고 말하기보다 통제할 수 있게 한다

“안전합니다”라는 문구만으로는 충분하지 않다. 사용자가 실제로 공개 범위, 차단, 신고, 대화 시작 여부를 직접 선택할 수 있어야 한다.

## 3.2 불안을 과하게 자극하지 않는다

허위 프로필, 기혼자, 비매너, 개인정보 유출 가능성을 지나치게 강조하면 사용자는 서비스 자체를 위험하게 느낄 수 있다. 문제를 숨기지는 않되, 대응 체계와 통제 수단을 함께 보여준다.

## 3.3 중요한 정보는 짧고 명확하게 안내한다

안전 관련 안내는 감성적 은유보다 명확성이 중요하다. 특히 개인정보, 신고, 차단, 인증 실패, 제재 안내에서는 직관적인 문장을 사용한다.

## 3.4 신고와 차단은 쉽게 접근 가능해야 한다

사용자가 불편함을 느끼는 순간 신고/차단 기능을 찾기 어려우면 안전감은 크게 떨어진다. 프로필, 추천 카드, 대화방, 매칭 이후 화면에서 접근 가능해야 한다.

## 3.5 상대를 의심하게만 만들지 않는다

프롤로그의 목표는 사용자가 모두를 의심하게 만드는 것이 아니라, 덜 의심하고 신중하게 시작할 수 있게 하는 것이다. 인증 배지와 추천 이유는 상대를 평가하는 도구가 아니라 신뢰의 출발점으로 보여야 한다.

## 3.6 여성 사용자의 안전감과 통제감을 우선 고려한다

초기 신뢰 형성에서 여성 사용자의 안전감은 전체 서비스 생태계의 품질에 직접적인 영향을 준다. 사진 공개, 대화 시작, 신고/차단, 정보 노출에 대한 통제감을 명확히 제공해야 한다.

# 4. 안전 UX가 필요한 주요 순간

| 단계 | 사용자가 느낄 수 있는 불안 | 필요한 UX |
|---|---|---|
| 랜딩페이지 | 믿을 수 있는 서비스인지 모르겠음 | 인증/신고/공개 범위 원칙 요약 |
| 가입/온보딩 | 개인정보를 많이 요구하는 것 같음 | 수집 목적과 공개 여부 안내 |
| 인증 | 원본 정보가 노출될까 걱정됨 | 인증 결과만 배지로 표시 안내 |
| 프로필 작성 | 내 정보가 어디까지 보일지 걱정됨 | 공개 미리보기, 민감 정보 안내 |
| 추천 카드 | 상대가 진짜인지 궁금함 | 인증 배지, 관계 목적 요약 |
| 매칭 리포트 | 추천 근거가 불투명함 | 추천 이유와 정보 사용 범위 안내 |
| 관심/거절 | 내 반응이 상대에게 공개될까 걱정됨 | 관심/거절 공개 기준 안내 |
| 대화 | 무례한 대화나 연락처 요구가 걱정됨 | 매너 안내, 신고/차단 접근성 |
| 만남 이후 | 노쇼/비매너 대응이 궁금함 | 사후 신고/피드백 |
| 탈퇴/차단 | 내 정보가 남을까 걱정됨 | 삭제/보관 기준 안내 |

# 5. 랜딩페이지 안전 UX

## 5.1 목표

랜딩페이지에서는 “프롤로그는 일반 소개팅 앱보다 더 신중하게 운영된다”는 인상을 줘야 한다. 다만 안전 장치를 너무 무겁게 강조하면 결정사나 심사 서비스처럼 느껴질 수 있으므로 균형이 필요하다.

## 5.2 노출해야 할 안전 요소

- 본인 인증
- 얼굴 인증
- 관계 목적 확인
- 싱글 상태 서약
- 인증 배지
- 신고/차단/제재 정책
- 공개 범위 설정

## 5.3 권장 문구

> 신뢰할 수 있는 만남을 위해 본인 인증과 얼굴 인증을 기본으로 합니다.

> 관계 목적과 만남 태도를 확인해, 가볍지 않은 만남을 원하는 사람들을 연결합니다.

> 신고와 차단 기능으로 불편한 경험을 줄이고, 허위 정보와 비매너 행동은 운영 기준에 따라 관리합니다.

## 5.4 피해야 할 문구

- 위험한 사용자를 완벽히 차단합니다.
- 100% 검증된 사람만 만날 수 있습니다.
- 기혼자와 사기꾼은 절대 없습니다.
- 프롤로그가 모든 만남을 책임집니다.

# 6. 온보딩 안전 UX

## 6.1 목표

온보딩에서는 사용자가 인증과 정보 입력을 “귀찮은 절차”가 아니라 “신뢰 가능한 추천을 위한 과정”으로 이해하게 해야 한다.

## 6.2 인증 전 안내 구조

각 인증 단계에는 다음 정보가 표시되어야 한다.

1. 왜 필요한가
2. 상대에게 공개되는가
3. 얼마나 걸리는가
4. 완료하면 어떤 이점이 있는가

## 6.3 본인 인증 문구

권장:

> 본인 인증은 가짜 계정과 중복 가입을 줄이기 위한 과정입니다.  
> 인증에 사용된 정보는 다른 사용자에게 공개되지 않습니다.

## 6.4 얼굴 인증 문구

권장:

> 얼굴 인증은 도용 사진을 줄이기 위한 과정입니다.  
> 인증 이미지는 상대에게 공개되지 않고, 프로필에는 인증 완료 배지만 표시됩니다.

## 6.5 관계 목적 설문 문구

권장:

> 몇 가지 질문으로 어떤 속도와 온도의 만남을 원하는지 확인합니다.  
> 답변은 더 잘 맞는 추천을 위해 사용됩니다.

## 6.6 싱글 상태 서약 문구

권장:

> 프롤로그는 현재 새로운 관계를 시작할 수 있는 싱글 사용자를 위한 서비스입니다.  
> 신뢰 있는 만남을 위해 싱글 상태 서약이 필요합니다.

## 6.7 온보딩 안전 체크포인트

- 인증 원본 비공개 안내가 있는가?
- 공개되는 정보와 비공개 정보가 구분되어 있는가?
- 사용자가 다음 단계에서 무엇을 해야 하는지 명확한가?
- “심사받는 느낌”보다 “신뢰를 함께 만드는 느낌”인가?
- 이탈이 큰 민감 단계에는 스킵/나중에 하기 또는 설명 보강이 있는가?

# 7. 프로필 작성 안전 UX

## 7.1 목표

사용자가 자신의 이야기를 충분히 표현하되, 불필요하게 민감한 정보를 공개하지 않도록 돕는다.

## 7.2 프로필 공개 미리보기

프로필 작성 후에는 “상대에게 보이는 내 프로필”을 확인할 수 있어야 한다.

필수 표시:

- 공개됨
- 비공개
- 인증 배지로만 표시됨
- 추천에만 사용됨

## 7.3 민감 정보 입력 안내

사용자가 자기소개나 문답에 연락처, SNS, 상세 주소, 회사명 등 민감 정보를 입력할 경우 안내한다.

예시:

> 연락처나 상세 주소처럼 개인을 직접 식별할 수 있는 정보는 프로필에 공개하지 않는 것을 권장합니다.

## 7.4 사진 공개 UX

사진은 안전감에 큰 영향을 준다.

MVP 권장 방향:

- 대표 사진은 추천 카드에 표시
- 추가 사진은 사용자가 공개 범위를 선택할 수 있도록 검토
- 얼굴 인증 사진은 절대 프로필에 자동 공개하지 않음
- 사진 업로드 전 공개 범위 안내

## 7.5 프로필 작성 체크포인트

- 사용자가 작성한 내용이 상대에게 보이는지 알 수 있는가?
- 인증에 사용한 자료가 프로필에 공개되지 않는다고 안내했는가?
- 민감 정보 입력을 예방하는 안내가 있는가?
- 공개 미리보기가 있는가?
- 사용자가 정보 공개를 조절할 수 있는가?

# 8. 추천 카드 안전 UX

## 8.1 목표

추천 카드에서는 상대를 신뢰할 수 있는 기본 근거를 제공하되, 사람을 심사표처럼 보이게 해서는 안 된다.

## 8.2 노출 요소

- 대표 사진
- 나이
- 지역/생활권
- 직업군
- 인증 배지
- 관계 목적 요약
- 라이프스타일 키워드
- 큐레이터의 메모
- 신고/차단 메뉴

## 8.3 인증 배지 UX

인증 배지는 과도한 등급화가 아니라 신뢰 장치로 보여야 한다.

권장 배지:

- 본인 인증 완료
- 얼굴 인증 완료
- 관계 목적 확인
- 싱글 상태 서약
- 직업/재직 인증

피해야 할 배지:

- 상위 1% 검증
- 프리미엄 스펙
- 고급 회원
- 완벽 인증
- 결혼 적합도 우수

## 8.4 추천 카드 신고 접근

추천 카드에는 작게라도 “신고하기” 또는 더보기 메뉴가 있어야 한다.

신고 사유 후보:

- 사진이 부적절해요
- 프로필이 허위 같아요
- 기혼자/연인 있음이 의심돼요
- 불쾌한 내용이 있어요
- 기타

# 9. 매칭 리포트 안전 UX

## 9.1 목표

매칭 리포트는 추천의 납득도를 높이는 동시에, 사용자가 상대를 과신하지 않도록 균형 있게 설계해야 한다.

## 9.2 권장 구조

1. 잘 맞을 수 있는 지점
2. 관계 성향의 비슷한 부분
3. 라이프스타일 공통점
4. 확인해보면 좋은 차이점
5. 첫 대화 추천 주제
6. 정보 사용 안내

## 9.3 안전한 추천 표현

권장:

> 두 분은 관계를 천천히 시작하는 편이고, 주말을 보내는 방식이 비슷해요.

> 다만 연락 빈도에 대한 기대는 조금 다를 수 있어요. 대화를 시작한다면 서로 편한 연락 방식을 가볍게 이야기해보세요.

비권장:

> 두 분은 완벽하게 잘 맞아요.

> 이 사람은 놓치면 안 되는 상대예요.

> 결혼까지 생각해볼 만한 상대예요.

## 9.4 정보 사용 안내

매칭 리포트 하단에는 정보 출처를 짧게 안내할 수 있다.

> 이 추천은 프로필, 관계 목적, 라이프스타일 응답, 인증 상태를 바탕으로 생성되었습니다.

# 10. 관심/거절 안전 UX

## 10.1 관심 보내기

관심 보내기는 사용자가 부담스럽지 않게 선택할 수 있어야 한다.

권장 문구:

- 관심 보내기
- 책갈피 남기기
- 이 사람과 대화해보고 싶어요

주의:

- “좋아요”만 사용할 경우 기존 소개팅 앱처럼 가볍게 느껴질 수 있다.
- “책갈피”는 브랜드 표현으로 좋지만, 기능 이해를 위해 “관심 보내기”와 병기하는 것이 좋다.

## 10.2 거절/넘기기

거절 사유는 추천 개선에 중요하지만, 사용자가 죄책감을 느끼거나 피로해지지 않게 해야 한다.

권장 문구:

> 더 나은 추천을 위해 가장 가까운 이유 하나만 알려주세요.  
> 이 내용은 상대에게 공개되지 않아요.

## 10.3 거절 사유 선택지

- 사진/분위기 호감이 낮아요
- 관계 목적이 달라 보여요
- 거리나 생활권이 맞지 않아요
- 대화 스타일이 맞지 않을 것 같아요
- 직업/생활 패턴이 맞지 않아요
- 취향이나 가치관이 달라 보여요
- 프로필 정보가 부족해요
- 기타

---

# 11. 대화 안전 UX

## 11.1 목표

대화는 프롤로그에서 실제 관계가 시작되는 핵심 지점이다. 사용자는 대화를 시작하면서 기대와 동시에 불안을 느낄 수 있다.

대화 안전 UX는 다음을 목표로 한다.

1. 대화 전 기본 매너를 상기시킨다.
2. 연락처/SNS 공유를 서두르지 않게 한다.
3. 불편한 대화에서 즉시 벗어날 수 있게 한다.
4. 신고/차단 기능을 쉽게 찾을 수 있게 한다.
5. 첫 대화가 어색하지 않도록 안전한 주제를 제안한다.

## 11.2 대화 시작 전 안내

상호 관심 후 첫 대화 진입 시 짧은 안내를 제공한다.

예시:

> 두 사람의 첫문장이 이어졌어요.  
> 서로의 속도를 존중하며 대화를 시작해보세요.

하단 보조 안내:

> 불편한 대화는 언제든 신고하거나 차단할 수 있습니다.

## 11.3 첫 대화 주제 추천

안전한 첫 대화 주제는 상대의 프로필과 매칭 리포트에서 가져온다.

예시:

- 주말을 보내는 방식이 비슷해요. 최근 가장 편안했던 주말은 어땠는지 물어보세요.
- 두 분 모두 천천히 알아가는 관계를 선호해요. 부담 없는 인사로 시작해보세요.
- 공통 관심사가 있어요. 프로필에 적힌 취향에 대해 가볍게 물어보세요.

## 11.4 위험 행동 예방 문구

대화방 내 안내 또는 정책 툴팁:

- 연락처나 SNS 공유는 충분히 신뢰가 생긴 뒤에 결정하세요.
- 금전 거래, 투자 권유, 외부 링크 이동은 주의해주세요.
- 불편한 메시지는 신고할 수 있으며, 신고 내용은 상대에게 공개되지 않습니다.

## 11.5 대화방 신고/차단 위치

대화방 상단 더보기 메뉴에 다음 기능을 포함한다.

- 신고하기
- 차단하기
- 매칭 해제하기
- 대화방 나가기
- 안전 가이드 보기

---

# 12. 신고/차단 안전 UX

## 12.1 목표

신고/차단은 사용자가 불편한 경험을 멈추는 핵심 기능이다. 사용자는 신고 과정에서 더 큰 부담을 느끼지 않아야 한다.

## 12.2 신고 UX 원칙

- 신고 사유를 먼저 선택하게 한다.
- 상세 입력은 선택으로 둔다.
- 가능하면 대화/프로필 정보가 자동으로 첨부되도록 한다.
- 신고 내용이 상대에게 공개되지 않음을 명확히 안내한다.
- 신고 후 차단 옵션을 제공한다.

## 12.3 신고 화면 예시

제목:

> 어떤 문제가 있었나요?

보조 문구:

> 신고 내용은 상대에게 공개되지 않으며, 프롤로그 운영 기준에 따라 검토됩니다.

선택지:

- 허위 프로필/도용 의심
- 기혼자/연인 있음 의심
- 부적절한 대화
- 개인정보 침해
- 외부 서비스/금전 유도
- 노쇼/비매너
- 기타

## 12.4 신고 완료 화면

예시:

> 신고가 접수되었습니다.  
> 불편한 경험을 알려주셔서 감사합니다. 프롤로그 운영 기준에 따라 검토하겠습니다.

버튼:

- 상대 차단하기
- 홈으로 돌아가기

## 12.5 차단 화면 예시

> 이 사용자를 차단하면 서로의 프로필과 대화가 보이지 않으며, 앞으로 추천에도 표시되지 않습니다.

버튼:

- 차단하기
- 신고하고 차단하기
- 취소

# 13. 여성 사용자 안전감 강화 UX

프롤로그는 남녀 모두를 위한 서비스지만, 초기 신뢰 형성에서는 여성 사용자의 안전감과 통제감을 특히 중요하게 본다.

## 13.1 핵심 요구

- 내 사진과 정보가 무분별하게 노출되지 않을 것
- 상대가 기본 인증을 완료했을 것
- 관계 목적이 가볍지 않을 것
- 신고/차단이 실제로 작동할 것
- 대화에서 불편하면 쉽게 빠져나올 수 있을 것

## 13.2 UX 반영

1. 정보 공개 미리보기 제공
2. 사진 공개 범위 안내
3. 인증 배지 명확히 표시
4. 싱글 상태 서약 표시
5. 대화방 신고/차단 상시 접근
6. 연락처/SNS 공유 주의 안내
7. 신고 후 상대에게 내용 비공개 안내
8. 매칭 전 상대 프로필 충분히 확인 가능

## 13.3 권장 문구

> 이 정보는 상대에게 바로 공개되지 않아요. 공개 범위는 단계별로 조정할 수 있어요.

> 불편한 대화는 언제든 차단할 수 있고, 신고 내용은 상대에게 공개되지 않습니다.

---

# 14. 남성 사용자 안전감/효율 UX

남성 사용자에게도 안전과 신뢰는 중요하다. 다만 주요 니즈는 “진지한 상대에게 닿는 효율”과 “인증/프로필 완성도의 보상”으로 나타날 수 있다.

## 14.1 핵심 요구

- 내가 성실하게 작성한 프로필이 제대로 반영될 것
- 무의미한 좋아요 경쟁이 줄어들 것
- 왜 추천됐는지 이해할 수 있을 것
- 인증을 완료하면 더 좋은 추천 경험을 받을 수 있을 것
- 부당한 신고나 오해에 대한 기준이 있을 것

## 14.2 UX 반영

1. 프로필 완성도 가이드
2. 인증 완료 보상 안내
3. 추천 이유 제공
4. 거절 사유가 상대에게 공개되지 않지만 추천 개선에 쓰인다는 안내
5. 제재 정책의 기본 기준 안내
6. 대화 매너 가이드 제공

## 14.3 권장 문구

> 프로필을 더 채우면 관계 성향이 맞는 사람에게 추천될 가능성이 높아져요.

> 인증과 프로필 완성도는 더 신뢰할 수 있는 추천 경험에 반영됩니다.

# 15. 안전 관련 화면에서 메타포 사용 기준

책방/서점 메타포는 프롤로그의 브랜드 자산이지만, 안전 관련 화면에서는 사용 강도를 낮춰야 한다.

| 화면 | 메타포 사용 | 원칙 |
|---|---:|---|
| 추천 카드 | 높음 | 큐레이터 메모 가능 |
| 매칭 리포트 | 높음 | 함께 읽어볼 이유 가능 |
| 대화 시작 | 중간 | 첫문장 표현 가능 |
| 인증 안내 | 낮음 | 명확한 설명 우선 |
| 개인정보 안내 | 매우 낮음 | 공개/비공개 명확히 표시 |
| 신고/차단 | 매우 낮음 | 감성보다 정확성 우선 |
| 제재 안내 | 없음 | 정책 문구 우선 |
| 탈퇴/삭제 | 없음 | 보관/삭제 기준 명확히 |

# 16. 안전 UX 문구 가이드

## 16.1 권장 문구

- 이 정보는 상대에게 공개되지 않습니다.
- 인증 결과만 배지로 표시됩니다.
- 공개 범위는 직접 설정할 수 있습니다.
- 신고 내용은 상대에게 공개되지 않습니다.
- 차단하면 서로의 프로필과 대화가 보이지 않습니다.
- 연락처나 SNS 공유는 충분히 신뢰가 생긴 뒤 결정하세요.
- 불편한 대화는 언제든 신고하거나 차단할 수 있습니다.

## 16.2 피해야 할 문구

- 프롤로그가 완벽하게 지켜드립니다.
- 100% 안전한 만남입니다.
- 걱정하지 마세요, 아무 일도 생기지 않습니다.
- 검증된 사람만 있으니 안심하세요.
- 책방지기가 몰래 보호해드릴게요.

# 17. 안전 UX 체크리스트

## 17.1 온보딩

- 정보 수집 목적이 명확한가?
- 상대 공개 여부가 표시되어 있는가?
- 인증 원본 비공개 안내가 있는가?
- 사용자가 부담보다 신뢰를 느끼는가?

## 17.2 프로필

- 공개 미리보기가 있는가?
- 민감 정보 입력을 예방하는가?
- 사진 공개 범위를 이해할 수 있는가?
- 인증 배지가 과도한 등급처럼 보이지 않는가?

## 17.3 추천/매칭

- 상대의 신뢰 근거가 보이는가?
- 추천 이유가 납득되는가?
- 신고/차단 메뉴가 접근 가능한가?
- 추천이 보장처럼 표현되지 않는가?

## 17.4 대화

- 대화 시작 전 매너 안내가 있는가?
- 신고/차단이 쉽게 보이는가?
- 연락처/SNS 공유 주의가 있는가?
- 첫 대화 주제가 안전하고 부담 없게 제안되는가?

## 17.5 신고/차단

- 신고 내용 비공개가 안내되는가?
- 신고 후 차단할 수 있는가?
- 차단 효과가 명확히 안내되는가?
- 신고 과정이 과도하게 길지 않은가?

## 17.6 탈퇴/삭제

- 탈퇴 시 삭제/보관 기준이 안내되는가?
- 신고/제재 기록의 예외 보관 가능성이 설명되는가?
- 사용자가 자신의 정보를 통제할 수 있다고 느끼는가?
