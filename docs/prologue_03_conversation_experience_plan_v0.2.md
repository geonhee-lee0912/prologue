# 1. 문서 목적

이 문서는 프롤로그에서 상호 관심 이후 사용자가 어떻게 대화를 시작하고, 이어가고, 필요할 때 안전하게 중단할 수 있는지를 정의하기 위한 문서다.

프롤로그의 대화 경험은 일반 소개팅 앱의 채팅 기능과 달라야 한다. 단순히 메시지를 주고받는 공간이 아니라, 추천 이유와 프로필 정보를 바탕으로 서로의 이야기를 자연스럽게 이어가는 공간이어야 한다. 동시에 비매너, 불쾌한 대화, 집요한 연락, 개인정보 요구 등 안전 리스크를 줄이는 장치가 필요하다.

이 문서는 다음 목적을 가진다.

1. 상호 관심 이후 대화 시작 흐름을 정의한다.
2. 첫 대화 진입 장벽을 낮추는 장치를 설계한다.
3. 프로필과 매칭 리포트를 대화 소재로 연결한다.
4. 대화 매너와 안전 장치를 정의한다.
5. 신고/차단/대화 종료 정책과 연결되는 UX 기준을 만든다.
6. MVP에서 구현할 대화 기능 범위를 정한다.

---

# 2. 대화 경험 한 줄 정의

**프롤로그의 대화 경험은 상호 관심이 생긴 두 사람이 추천 이유와 서로의 이야기를 바탕으로 부담 없이 첫 대화를 시작하고, 안전하게 관계 가능성을 확인하도록 돕는 경험이다.**

브랜드 문장으로는 다음과 같이 표현할 수 있다.

> 첫문장이 이어졌어요. 이제 서로의 이야기를 대화로 이어갈 수 있습니다.

# 3. 대화 경험 설계 원칙

## 3.1 대화는 상호 관심 이후에만 열린다

프롤로그는 무분별한 메시지 발송을 허용하지 않는다. 한쪽의 일방적 메시지가 아니라, 양쪽이 관심을 표현한 이후 대화가 가능해야 한다.

## 3.2 첫 메시지 부담을 낮춘다

좋은 추천이 있어도 첫 메시지가 어렵다면 대화가 시작되지 않는다. 프롤로그는 프로필, 매칭 리포트, 공통 관심사에서 대화 주제를 추천해 첫 문장을 돕는다.

## 3.3 대화는 추천 이유와 연결된다

프롤로그의 차별점은 “왜 추천됐는지”를 알고 대화를 시작할 수 있다는 점이다. 대화방에서는 매칭 리포트의 핵심 이유와 첫 대화 키워드를 쉽게 다시 볼 수 있어야 한다.

## 3.4 안전과 매너를 전제로 한다

대화 시작 전과 대화 중에 매너 안내, 신고, 차단, 공개 범위 안내가 명확해야 한다. 안전 장치는 숨겨진 기능이 아니라 사용자가 필요할 때 바로 찾을 수 있어야 한다.

## 3.5 과한 낭만화와 압박을 피한다

“운명”, “놓치면 후회”, “빨리 답장” 같은 압박형 문구를 쓰지 않는다. 프롤로그의 대화는 신중하고 존중하는 분위기를 유지해야 한다.

# 4. 대화 전체 흐름

MVP 기준 대화 흐름은 다음과 같다.

> 관심 보내기 → 상대도 관심 보내기 → 상호 관심 발생 → 매칭 완료 안내 → 대화 매너 안내 → 첫 대화 주제 선택 또는 직접 입력 → 대화 시작 → 대화 중 프로필/리포트 참고 → 신고/차단/대화 종료 가능

---

# 5. 상호 관심 발생

## 5.1 발생 조건

대화는 다음 조건이 모두 충족될 때 열린다.

1. 사용자 A가 사용자 B에게 관심을 보냄
2. 사용자 B도 사용자 A에게 관심을 보냄
3. 두 사용자 모두 필수 인증 상태가 유효함
4. 두 사용자 중 한쪽이 차단 또는 이용 제한 상태가 아님

## 5.2 매칭 완료 화면

### 화면 목적

상호 관심이 발생했음을 알려주고, 대화 시작으로 자연스럽게 이동시킨다.

### 주요 구성

- 매칭 완료 메시지
- 상대의 기본 정보 요약
- 추천 이유 요약
- 첫 대화 주제 보기
- 대화 시작하기 CTA
- 안전 안내 링크

### 카피 예시

> 첫문장이 이어졌어요.  
> 이제 서로의 이야기를 대화로 이어갈 수 있습니다.

보조 문구:

> 두 분이 관심을 함께 남겨 대화가 열렸어요. 대화를 시작하기 전, 서로를 존중하는 매너를 확인해주세요.

CTA:

- 대화 시작하기
- 첫 대화 주제 보기

# 6. 대화 매너 안내

## 6.1 목적

대화 시작 전 매너 기준을 짧고 명확하게 안내한다. 이는 사용자를 겁주기 위한 장치가 아니라, 프롤로그의 신뢰 있는 대화 문화를 만드는 장치다.

## 6.2 노출 시점

- 첫 대화방 진입 전 1회
- 신고/차단 기능 안내 시
- 정책 변경 시
- 반복적인 부적절 메시지 감지 시, 향후 기능

## 6.3 안내 내용

MVP 기준 매너 안내는 다음 항목을 포함한다.

1. 상대의 답장 속도를 존중한다.
2. 연락처, 외부 SNS, 만남 요청은 대화가 충분히 이어진 뒤 자연스럽게 제안한다.
3. 성적, 모욕적, 위협적, 집요한 메시지는 제재될 수 있다.
4. 불편한 대화는 언제든 신고하거나 차단할 수 있다.

### 카피 예시

> 프롤로그의 대화는 서로의 속도를 존중하는 것에서 시작합니다.  
> 불편한 대화는 언제든 신고하거나 차단할 수 있어요.

체크 문구:

> 서로를 존중하는 대화 매너에 동의합니다.

# 7. 첫 대화 주제 추천

## 7.1 목적

첫 메시지 부담을 줄이고, 대화가 프로필과 추천 이유에서 자연스럽게 시작되도록 돕는다.

## 7.2 생성 기준

첫 대화 주제는 다음 정보에서 생성한다.

- 상대의 나의 프롤로그
- 이야기의 목차 답변
- 관계의 문체
- 공통 라이프스타일 키워드
- 매칭 리포트의 잘 맞는 지점
- 사용자가 공개한 취향 키워드

## 7.3 주제 유형

| 유형 | 설명 | 예시 |
|---|---|---|
| 공통 취향형 | 공통 관심사 기반 | 두 분 모두 주말 산책을 좋아한다고 답했어요. 좋아하는 산책 코스를 물어보면 어떨까요? |
| 생활 리듬형 | 주말/휴식/일상 기반 | 둘 다 조용한 휴식을 선호해요. 쉬는 날을 어떻게 보내는지 물어보세요. |
| 관계 성향형 | 대화/연락/관계 속도 기반 | 두 분 모두 천천히 알아가는 관계를 선호해요. 편안한 대화부터 시작해보세요. |
| 프로필 문장형 | 자기소개 문장 기반 | 상대가 요즘 가장 즐겁게 하는 일을 적어두었어요. 그 이야기를 이어가보세요. |
| 가벼운 질문형 | 부담 없는 시작 질문 | 오늘 하루 중 가장 편안했던 순간은 언제였나요? |

## 7.4 첫 메시지 템플릿

사용자는 추천된 주제를 그대로 보내거나 수정해서 보낼 수 있다.

예시:

- 프로필에 주말 산책을 좋아한다고 적어두셨더라고요. 자주 가는 산책 코스가 있으세요?
- 저도 천천히 알아가는 관계를 선호하는 편이라 반가웠어요. 보통 처음 대화할 때 어떤 분위기를 좋아하세요?
- 큐레이터 메모에서 주말을 보내는 방식이 비슷하다고 해서 궁금했어요. 쉬는 날은 주로 어떻게 보내세요?

## 7.5 주의할 점

첫 대화 주제는 사적인 정보나 민감한 조건을 바로 묻지 않아야 한다.

비권장 질문:

- 연봉이 어떻게 되세요?
- 결혼은 언제 하고 싶으세요?
- 집은 자가인가요?
- 왜 아직 싱글이세요?
- 바로 만날 수 있나요?

# 8. 대화방 정보 구조

## 8.1 상단 영역

- 상대 프로필 사진 또는 대표 이미지
- 이름 또는 닉네임
- 나이/지역 요약
- 신뢰 배지 요약
- 프로필 보기
- 신고/차단 메뉴

## 8.2 매칭 맥락 영역

대화방 상단 또는 접힘 영역에 다음 내용을 제공한다.

- 첫문장이 이어졌어요
- 잘 맞는 지점 1~2개
- 첫 대화 추천 주제
- 매칭 리포트 다시 보기

## 8.3 메시지 영역

- 텍스트 메시지
- 시간 표시
- 읽음 여부, MVP에서는 선택
- 시스템 안내 메시지
- 신고 가능한 메시지 단위 메뉴, MVP에서는 대화방 단위 신고 우선 가능

## 8.4 하단 입력 영역

- 메시지 입력창
- 보내기 버튼
- 첫 대화 주제 불러오기
- 사진/파일 전송은 MVP에서 제외 권장

# 9. 대화방 운영 정책

## 9.1 기본 원칙

프롤로그의 내부 대화방은 무한정 열어두지 않는다.

프롤로그는 무한 스와이프와 무제한 탐색을 지양하는 서비스이므로, 대화 역시 방치형 오픈 채팅방이 아니라 **관계 가능성을 확인하고 다음 단계로 자연스럽게 이동하도록 돕는 제한된 대화 공간**이어야 한다.

대화방 제약의 목적은 사용자를 막는 것이 아니라 다음 가치를 지키기 위함이다.

1. 대화방 방치와 잠수를 줄인다.
2. 한쪽이 원하지 않는 대화가 오래 지속되는 것을 막는다.
3. 사용자가 대화 지속 여부를 주기적으로 선택하게 한다.
4. 내부 채팅 운영, 신고, 보관, 삭제 부담을 관리한다.
5. 충분히 대화한 뒤 상호 동의로 연락처 교환 또는 외부 대화 전환을 유도한다.

## 9.2 권장 대화 기간 정책

MVP 기준 대화방은 다음 구조를 권장한다.

| 항목 | 정책 |
|---|---|
| 대화방 오픈 조건 | 상호 관심 발생 |
| 기본 대화 기간 | 7일 |
| 연장 조건 | 양쪽 모두 연장 동의 |
| 1회 연장 기간 | 7일 |
| MVP 최대 유지 기간 | 무료 14일, Plus 21일 |
| 정식 서비스 검토안 | 무료 21일, Plus 30일 |
| 기간 종료 후 | 대화 종료, 연락처 교환 요청, 신고/차단 중 선택 |

기본 대화 기간은 7일로 둔다. 7일은 첫 인사, 몇 차례의 대화, 관계 가능성 판단을 하기에 충분하면서도 대화를 무기한 방치하지 않도록 만드는 기준이다.

대화 연장은 한쪽의 일방 요청만으로는 이루어지지 않는다. 양쪽 모두가 “대화를 더 이어가고 싶다”고 선택했을 때만 7일 단위로 연장된다.

## 9.3 대화 종료 전 안내

대화방 종료 24시간 전에는 사용자에게 선택지를 안내한다.

### 카피 예시

> 대화 시간이 곧 종료돼요.  
> 더 이어가고 싶다면 대화 연장이나 연락처 교환을 요청할 수 있어요.

CTA:

- 7일 더 대화하기
- 연락처 교환 요청하기
- 자연스럽게 종료하기

대화 종료 안내는 불안을 자극하거나 결제를 압박하는 방식이 아니어야 한다. “놓치면 후회해요”, “지금 결제하지 않으면 대화가 사라져요” 같은 표현은 사용하지 않는다.

## 9.4 연락처 교환 정책

프롤로그는 연락처를 자동으로 공개하지 않는다. 연락처 교환은 반드시 **상호 동의형**으로 설계한다.

권장 플로우는 다음과 같다.

> 대화 기간 종료 전 또는 대화 중 충분한 교류 발생  
> → 한쪽이 연락처 교환 요청  
> → 상대가 수락  
> → 양쪽 연락처 또는 외부 대화 링크 공개  
> → 서비스 내 대화방은 유지 기간 종료 후 보관/종료 상태로 전환

연락처 교환 화면에는 다음 안내가 필요하다.

> 연락처는 상대가 수락한 경우에만 서로에게 공개됩니다.  
> 외부 대화로 이동한 이후의 대화 내용은 프롤로그가 직접 확인할 수 없으므로, 불편한 상황이 생기면 신고 화면에서 관련 내용을 제출해주세요.

## 9.5 메시지 수 제한 여부

메시지 수 자체는 제한하지 않는다.

“하루 10개 메시지”처럼 메시지 개수를 제한하면 대화가 인위적으로 끊기고, 진지한 관계를 확인하려는 사용자에게 불편을 줄 수 있다. 프롤로그는 메시지 수를 막기보다 **대화 기간, 연장 동의, 연락처 교환, 신고/차단**으로 대화 품질을 관리한다.

비추천 정책:

- 하루 메시지 개수 제한
- 무료 사용자는 답장 횟수 제한
- 매칭 후 답장하려면 결제 필요
- 유료 사용자만 대화 가능

특히 “상호 매칭 후 답장하려면 결제” 구조는 사용자가 속았다고 느낄 수 있으므로 사용하지 않는다.

## 9.6 무료/유료 사용자별 대화 차이

무료 사용자도 상호 매칭 후 기본 대화는 가능해야 한다.

무료와 유료의 차이는 “대화 가능 여부”가 아니라, 대화를 더 잘 시작하고 이어가도록 돕는 보조 기능에서 만든다.

| 기능 | 무료 | Plus |
|---|---:|---:|
| 상호 매칭 후 기본 채팅 | 가능 | 가능 |
| 기본 대화 기간 | 7일 | 7일 |
| 대화 연장 | 양쪽 동의 시 가능 | 양쪽 동의 시 가능 |
| MVP 최대 유지 기간 | 14일 | 21일 |
| 첫 대화 주제 | 1~2개 제공 | 더 다양하게 제공 |
| 매칭 리포트 기반 질문 | 제한 | 제공 |
| 대화 분위기 리마인드 | 제한 | 제공 |
| 연락처 교환 요청 | 가능 | 가능 |
| 신고/차단 | 가능 | 가능 |

유료화는 사용자의 대화 권리를 막는 방식이 아니라, **대화의 시작과 지속을 돕는 품질 보조 기능**으로 설계한다.

## 9.7 내부 채팅 기능 제한

MVP의 내부 채팅은 텍스트 중심으로 시작한다.

MVP에서 허용하는 기능:

- 텍스트 메시지
- 기본 이모지
- 프롤로그 추천 질문 카드
- 매칭 리포트 요약 보기
- 대화 기간 표시
- 대화 연장 요청
- 연락처 교환 요청
- 신고/차단
- 대화 종료

MVP에서 제외하거나 보류하는 기능:

- 사진 전송
- 동영상 전송
- 음성 메시지
- 파일 전송
- 자유 링크 전송
- 위치 공유
- 읽음 확인
- 온라인 상태 표시

사진, 영상, 링크, 위치 공유는 안전 리스크와 운영 부담을 높인다. 읽음 확인과 온라인 상태 표시는 답장 압박을 만들 수 있으므로 MVP에서는 제외한다.

## 9.8 대화 종료 후 피드백

대화가 종료되면 추천 품질 개선을 위해 짧은 피드백을 요청한다.

### 카피 예시

> 이번 대화는 종료되었어요.  
> 다음 추천을 더 잘 준비할 수 있도록 짧게 알려주세요.

피드백 선택지:

- 대화가 잘 이어졌어요
- 답장이 거의 없었어요
- 관계 목적이 달라 보였어요
- 대화 매너가 아쉬웠어요
- 외부 연락처 요청이 부담스러웠어요
- 신고가 필요해요

이 피드백은 상대에게 공개하지 않으며, 추천 품질 개선과 안전 정책 운영에 활용한다.

# 11. 대화 안전 정책 UX

## 10.1 신고하기

### 진입 위치

- 대화방 상단 메뉴
- 상대 프로필 메뉴
- 메시지 길게 누르기, MVP 이후

### 신고 사유 후보

- 무례하거나 모욕적인 메시지
- 성적 불쾌감을 주는 메시지
- 집요한 연락 또는 압박
- 허위 정보 의심
- 기혼자/연인 있음 의심
- 금전 요구 또는 사기 의심
- 외부 연락처 강요
- 기타

### 신고 완료 카피

> 신고가 접수되었습니다.  
> 신고한 내용은 상대에게 공개되지 않으며, 운영 정책에 따라 검토됩니다.

## 10.2 차단하기

### 차단 효과

- 서로의 프로필이 보이지 않음
- 대화방이 비활성화됨
- 상대가 새 메시지를 보낼 수 없음
- 차단 여부는 상대에게 직접 알림으로 전달되지 않음

### 차단 확인 카피

> 이 사용자를 차단할까요?  
> 차단하면 서로의 프로필과 대화가 보이지 않습니다.

## 10.3 대화 종료

대화 종료는 차단과 다르게, 사용자가 대화방을 정리하거나 숨기는 기능이다. MVP에서는 별도 기능으로 두지 않고, 대화방 숨김 또는 차단 기능으로 단순화할 수 있다.

# 12. 알림 설계

- 첫문장이 이어졌어요. 대화를 시작해보세요.
- 상대가 답장을 보냈어요.

비권장:

- 빨리 답장하지 않으면 놓칠 수 있어요!
- 상대가 기다리고 있어요!
- 지금 답장하지 않으면 관심이 식을 수 있어요!

## 11.2 알림 유형

| 알림 | 발송 조건 | 예시 |
|---|---|---|

| 첫 메시지 미발송 | 매칭 후 일정 시간 대화 없음 | 대화를 시작하기 좋은 주제를 준비했어요. |
| 새 메시지 | 상대가 메시지 전송 | 새로운 메시지가 도착했어요. |
| 대화 재개 | 일정 기간 대화 없음 | 이어서 이야기해볼 주제가 있어요. |
| 안전 안내 | 신고/차단 관련 | 불편한 대화는 언제든 신고하거나 차단할 수 있어요. |

# 13. 대화 데이터와 추천 개선

대화 경험에서 수집 가능한 데이터는 추천 품질과 안전 개선에 활용할 수 있다. 단, 민감도가 높으므로 수집 목적과 활용 범위를 명확히 해야 한다.

## 12.1 활용 가능 데이터

- 매칭 후 첫 메시지 전송 여부
- 첫 메시지까지 걸린 시간
- 대화 지속 일수
- 메시지 왕복 횟수
- 대화 중 프로필/리포트 재열람 여부
- 신고/차단 여부
- 대화 종료 여부

## 12.2 주의 데이터

메시지 본문은 민감한 개인정보와 사적 대화가 포함될 수 있다. 추천 알고리즘 학습에 직접 활용하는 경우에는 별도 동의, 비식별화, 보관 정책이 필요하다.

## 12.3 추천 개선 활용 예시

- 매칭은 되지만 대화 시작률이 낮은 추천 유형 파악
- 첫 대화 주제 사용률 분석
- 특정 거절 사유와 대화 중단률 비교
- 신고/차단이 잦은 사용자 패턴 분석
