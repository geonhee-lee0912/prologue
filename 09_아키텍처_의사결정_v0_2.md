## 2. 결정 근거 요약

### 2.1 왜 B안인가

v0.1 문서에서 8개 평가축으로 3안을 비교한 결과:

- **A안 (NestJS + 자체 인프라)**: 통제력은 가장 높지만 1인 운영에 인프라 부담이 과도함
- **B안 (NestJS + Supabase)**: 비즈니스 로직 명확성 + 인프라 부담 완화 + 한국 외부 서비스 적합성을 모두 확보
- **C안 (Vercel + Supabase)**: MVP 출시 속도는 가장 빠르지만 비즈니스 로직 분산, 본인인증 IP 이슈, 추천 배치 시간 제한이 부담

프롤로그 도메인의 특징:

1. 비즈니스 로직이 다단계 트랜잭션 (관심 보내기 → 상호 매칭 → 대화방 생성 → 시스템 메시지 → 푸시)
2. 개인정보 노출 비용이 큼 (거절 사유, 신고 내용, 점수)
3. 한국 외부 서비스 의존도 높음 (PASS/NICE 본인인증, NHN Toast SMS)
4. 추천 배치 같은 무거운 백그라운드 작업
5. 1인 운영 환경

이 중 1, 4번은 NestJS + Railway 의 지속 프로세스 환경이 유리하고, 2, 3번은 Supabase + Railway 조합이 유리하다. B안이 이 균형점이다.

### 2.2 왜 Vercel + Railway 를 함께 쓰는가

Vercel 과 Railway 는 다른 카테고리의 서비스다.

| | Vercel | Railway |
|---|---|---|
| 작동 방식 | 요청이 올 때만 실행 (serverless) | 24시간 켜져 있음 |
| 최적 용도 | Next.js, 정적 사이트, 짧은 API | 풀스택 백엔드, 장시간 작업 |
| 장시간 작업 | 불가 | 자유로움 |
| WebSocket | 불가 | 가능 |
| BullMQ 큐 | 불가 (지속 프로세스 필요) | 가능 |
| 고정 IP | 불가 (IP 동적) | 가능 |

NestJS 가 해야 할 일 (추천 배치, BullMQ 큐, 본인인증 콜백) 은 Vercel 에 안 맞는다. 하지만 운영자 도구(Next.js)는 Vercel 에 잘 맞는다. **두 서비스가 역할이 달라 양립한다.**

### 2.3 왜 Supabase Cloud 두 프로젝트인가

로컬 개발에서 Supabase Local CLI 를 쓰는 대안이 있지만 Docker 가 필요하다. 비개발자에게 Docker 운영은 진입 장벽이다.

대신 Supabase Cloud 에 dev 프로젝트와 prod 프로젝트를 따로 만든다.

- dev 프로젝트: 무료 티어
- prod 프로젝트: Pro 플랜 ($25/월)
- 두 프로젝트는 환경 변수만 바꾸면 전환됨
- 클로드 코드가 dev 에서 자유롭게 실험 가능, prod 는 안전

### 2.4 왜 Prisma + Supabase CLI 둘 다인가

| 도구 | 잘하는 영역 | 사용 |
|---|---|---|
| Prisma | 테이블 구조, 컬럼, 인덱스, 외래키 | `prisma migrate` 로 자동 SQL 생성 |
| Supabase CLI | RLS 정책, DB 함수, 트리거 | SQL 직접 작성 후 적용 |

RLS 는 SQL 로만 작성 가능하므로 Supabase CLI 가 필요하다. 테이블 구조는 Prisma 가 훨씬 편하다. 두 도구가 서로 다른 영역을 담당하므로 충돌하지 않는다.

운영 규칙: **Supabase Studio 에서 스키마를 직접 변경하지 않는다.** 모든 변경은 코드 저장소 → 마이그레이션 적용 흐름을 거친다.

## 5. 데이터 흐름 패턴 (4가지)

### 패턴 1: 단순 조회 — 모바일이 Supabase 직접 호출

내 프로필 조회, 매칭된 대화방 목록 같은 단순 조회는 모바일이 Supabase JS 클라이언트로 직접 호출한다. RLS 가 권한을 강제한다.

```
모바일 → Supabase (RLS 적용)
```

### 패턴 2: 비즈니스 트랜잭션 — NestJS 경유

관심 보내기, 매칭 생성, 신고 처리 같은 다단계 트랜잭션은 모바일이 NestJS API 를 호출한다. NestJS 가 service role key 로 Supabase 에 직접 쓴다.

```
모바일 → NestJS API → Supabase (service role)
```

### 패턴 3: 실시간 메시지 구독 — 모바일이 Supabase Realtime 직접 구독

대화방의 새 메시지 수신은 모바일이 Supabase Realtime 채널을 구독한다. RLS 가 본인이 참여한 대화만 구독을 허용한다.

```
모바일 ⇄ Supabase Realtime (RLS 적용)
```

### 패턴 4: 메시지 전송 — NestJS 경유

메시지 전송은 모바일이 직접 INSERT 하지 않고 NestJS API 를 거친다. 대화 만료, 차단, 금지 표현 검증을 백엔드가 보장한다.

```
모바일 → NestJS API → Supabase INSERT → Realtime 자동 broadcast → 다른 클라이언트
```

## 6. RLS 설계 원칙

### 6.1 모바일이 직접 SELECT 할 수 있는 테이블

RLS 필수. 정책 작성이 까다롭다.

- `profiles` (자신 또는 매칭/추천 관계가 있는 상대)
- `photos` (자신 또는 매칭/추천 관계가 있는 상대)
- `messages` (참여한 대화방만)
- `recommendations` (자신이 받은 것만, 점수 컬럼은 절대 노출 금지)
- `recommendation_reasons` (자신이 받은 추천에 대한 것만)
- `matches` (참여한 매칭만)
- `conversations` (참여한 대화만)
- `user_actions` (자신의 액션만, 다른 사용자 액션은 일체 비공개)

### 6.2 모바일이 SELECT 할 수 없는 테이블

운영자 또는 NestJS 만 접근.

- `user_actions.skip_reason` (거절 사유 컬럼은 RLS로 컬럼 자체를 가림)
- `reports` (모든 신고 정보)
- `blocks.blocked` 의 식별자 (차단 여부는 알리되 누가 했는지는 비공개)
- `admin_users`, `admin_reviews`, `admin_audit_logs`
- `payments`
- `refresh_tokens`
- `user_consents`
- `user_auths` (자신 것만 SELECT 허용, 인증 원본 컬럼은 절대 비공개)

### 6.3 모바일이 INSERT/UPDATE/DELETE 할 수 있는 테이블

거의 없음. 대부분 NestJS 경유.

- `profile_answers` (자신의 답변만, UPSERT)
- `photos` (자신의 사진만 업로드)
- 그 외 모든 변경은 NestJS API 경유
