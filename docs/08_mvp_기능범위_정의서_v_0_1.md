문서 상태: Draft  
작성 기준: 프롤로그 서비스 콘셉트 기획서 v1.1  
문서 목적: 프롤로그 MVP에서 반드시 개발할 기능, 후순위 기능, 제외할 기능을 명확히 정의한다.

---

# 1. 문서 목적

이 문서는 프롤로그 MVP에서 반드시 개발할 기능, 후순위 기능, 제외할 기능을 명확히 정의한다.

MVP의 목적은 모든 기능을 완성하는 것이 아니라, 다음 핵심 경험이 실제로 작동하는지 검증하는 것이다.

> 인증하고, 추천받고, 이유를 읽고, 관심을 보내고, 대화한다.

프롤로그는 초기부터 결혼중개나 결정사 대체 서비스로 포지셔닝하지 않는다. MVP는 **인증 기반 진지한 소개팅 앱**으로서의 핵심 흐름을 검증하는 데 집중한다.

---

# 2. MVP 핵심 목표

MVP는 다음 질문에 답해야 한다.

1. 사용자는 인증 기반 소개팅 앱에 가입할 의향이 있는가?
2. 사용자는 본인/얼굴 인증과 관계 목적 확인을 완료할 수 있는가?
3. 이야기형 프로필 작성이 부담스럽지만 감당 가능한 수준인가?
4. 제한된 추천 방식이 무한 탐색보다 매력적으로 느껴지는가?
5. 매칭 리포트와 큐레이터 메모가 추천 납득도를 높이는가?
6. 관심/거절 액션이 자연스럽게 발생하는가?
7. 상호 관심 이후 내부 대화방에서 대화가 시작되는가?
8. 신고/차단/연락처 교환 정책이 안전감을 주는가?

---

# 3. MVP 기본 전제

MVP 기능범위는 다음 전제를 기준으로 한다.

- 서비스명은 **프롤로그**다.
- 포지션은 **인증 기반 진지한 소개팅 앱**이다.
- 핵심 타깃은 **26~39세 싱글**이다.
- 핵심 가치는 **신뢰, 진정성, 큐레이션, 이야기형 프로필, 안전감**이다.
- 핵심 경험은 **인증 → 프로필 작성 → 관계 목적 확인 → 추천 카드 → 매칭 리포트 → 관심/거절 → 대화**다.
- 추천은 무한 스와이프가 아니라 **제한된 큐레이션 추천**이다.
- 프로필은 조건표가 아니라 **이야기형 프로필**이다.
- 상호 관심 이후에는 **내부 대화방**을 제공한다.
- 무료 사용자와 유료 사용자는 기본적으로 서로 매칭될 수 있다.
- 유료화의 차이는 매칭 가능 여부가 아니라 추천 수, 리포트 깊이, 대화 보조, 추가 추천권, 향후 고신뢰 풀 접근에서 만든다.
- 추가 추천권은 무제한 탐색권이 아니라 **추가 추천 카드 1개 열람권**이다.

# 5. MVP 필수 기능

## 5.1 회원가입/로그인

### 목적

사용자가 프롤로그 계정을 생성하고 서비스 이용을 시작할 수 있게 한다.

### 범위

- 휴대폰 번호 기반 가입 우선
- 간편 로그인은 선택 가능
- 약관 동의
- 개인정보 처리방침 동의
- 만 19세 미만 또는 서비스 대상 연령 외 사용자 제한

### 정책

- 동일 휴대폰 번호로 중복 가입할 수 없다.
- 탈퇴 후 재가입 제한 정책은 별도 검토한다.
- 가입 완료 후 인증 안내 화면으로 이동한다.

---

## 5.2 기본 인증

MVP에서 매칭을 받기 위해 필요한 기본 인증은 다음과 같다.

- 본인 인증
- 얼굴 인증
- 나이 확인
- 관계 목적 설문
- 매너 서약
- 싱글 상태 서약

### 인증 원칙

1. 인증은 강하게, 공개는 부드럽게 한다.
2. 인증 원본보다 인증 결과와 신뢰 배지를 중심으로 보여준다.
3. 민감한 정보는 공개 범위를 제한한다.
4. 필수 인증을 완료해야 추천을 받을 수 있다.
5. 인증 상태는 추천 품질과 사용자 신뢰에 영향을 준다.

---

## 5.3 직업/재직 인증

직업/재직 인증은 MVP에서 필수가 아니라 **강력 권장 인증**으로 둔다.

### MVP 적용 방식 후보

1. 수동 검수 기반 인증
2. 재직 증빙 업로드 후 운영자 확인
3. 인증 신청만 받고 실제 검증은 베타 운영 단계에서 처리

### 정책

- 인증 완료자는 프로필에 직업/재직 인증 배지를 표시할 수 있다.
- 인증 완료 여부는 추천 품질 또는 노출 우선순위에 반영할 수 있다.
- 민감한 회사명/증빙자료는 공개하지 않는다.

---

## 5.4 프로필 작성

### 목적

사용자가 조건표가 아니라 이야기형 프로필을 구성하도록 한다.

### 필수 입력

- 대표 사진
- 기본 정보
- 직업군
- 거주 지역
- 나의 프롤로그
- 관계 목적
- 관계의 문체
- 라이프스타일 키워드
- 대화 스타일

### 프로필 구성

1. **사진**
   - 대표 사진
   - 일상 사진
   - 취향이 드러나는 사진

2. **기본 정보**
   - 나이
   - 지역
   - 직업군
   - 인증 배지

3. **나의 프롤로그**
   - 나를 가장 잘 설명하는 문장
   - 요즘 가장 즐겁게 하는 일
   - 편안함을 느끼는 순간
   - 좋은 관계에서 중요하게 생각하는 것

4. **이야기의 목차**
   - 일과 커리어
   - 주말과 휴식
   - 친구와 가족
   - 취향과 관심사

5. **관계의 문체**
   - 연락 빈도
   - 대화 스타일
   - 관계 시작 속도
   - 갈등 해결 방식
   - 애정 표현 방식

### 정책

- 프로필 작성은 한 번에 모두 입력하도록 강제하지 않는다.
- 필수 항목과 추가 항목을 나눈다.
- 필수 항목을 완료해야 추천 대상이 된다.
- 프로필 완성도는 추천 품질에 반영할 수 있다.

---

## 5.5 추천 카드

### 목적

사용자가 제한된 수의 추천을 신중하게 읽어볼 수 있도록 한다.

### 구성

- 사진
- 나이
- 지역
- 직업군
- 인증 배지
- 관계 목적
- 대화 스타일
- 라이프스타일 키워드
- 큐레이터의 메모
- 관심 보내기
- 넘기기

### 정책

- 추천 수는 MVP 운영 가설로 제한한다.
- 추천 후보가 없을 경우 빈 상태 화면을 제공한다.
- 추천 후보가 부족할 때 억지 추천을 하지 않는다.
- 이미 넘기거나 관심을 보낸 상대는 같은 추천 세션에서 다시 노출하지 않는다.

---

## 5.6 매칭 리포트

### 목적

사용자가 왜 이 사람이 추천되었는지 납득할 수 있게 한다.

### 구성

- 잘 맞는 이유
- 비슷한 관계 성향
- 공통 라이프스타일
- 확인하면 좋은 차이점
- 첫 대화 추천 주제
- 프롤로그 큐레이터의 메모

### 정책

- MVP에서는 점수보다 설명형 문장을 우선한다.
- 추천 이유가 없는 추천은 제공하지 않는 것을 원칙으로 한다.
- 무료 사용자는 요약 리포트, Plus는 상세 리포트 방향을 가설로 둔다.

---

## 5.7 관심 보내기/넘기기

### 관심 보내기

- 사용자는 추천 상대에게 관심을 보낼 수 있다.
- 관심 표현은 내부 용어로 책갈피를 사용할 수 있으나, 버튼명은 명확성을 위해 “관심 보내기”를 우선한다.
- 상대도 관심을 보내면 상호 매칭이 발생한다.

### 넘기기

- 사용자는 추천 상대를 넘길 수 있다.
- 넘기기 선택 시 거절 사유 입력 모달을 표시한다.
- 거절 사유는 상대에게 공개되지 않는다.
- 거절 사유는 추천 개선 데이터로 활용한다.

### 거절 사유 선택지

- 사진/분위기 호감이 낮아요
- 관계 목적이 달라 보여요
- 거리나 생활권이 맞지 않아요
- 대화 스타일이 맞지 않을 것 같아요
- 직업/생활 패턴이 맞지 않아요
- 취향이나 가치관이 달라 보여요
- 프로필 정보가 부족해요
- 기타

---

## 5.8 상호 관심 매칭

### 목적

두 사용자가 서로 관심을 보냈을 때 대화 가능 상태로 전환한다.

### 정책

- A가 B에게 관심을 보내고, B도 A에게 관심을 보내면 상호 매칭이 발생한다.
- 상호 매칭 시 양쪽 사용자에게 알림을 보낸다.
- 매칭 완료 화면에서 안전 안내와 첫 대화 주제를 제공한다.
- 매칭과 동시에 내부 대화방을 생성한다.

---

## 5.9 내부 대화방

### 목적

상호 관심 이후 연락처를 바로 교환하지 않고, 앱 안에서 먼저 안전하게 대화할 수 있게 한다.

### MVP 범위

- 텍스트 메시지
- 시스템 메시지
- 첫 대화 주제 제공
- 대화 기간 표시
- 신고/차단
- 연락처 교환 요청

### MVP 제외

- 사진 전송
- 동영상 전송
- 음성 메시지
- 파일 전송
- 위치 공유
- 자유 링크 전송
- 읽음 확인
- 온라인 상태 표시

### 대화 기간 정책

- 기본 대화 기간은 7일이다.
- 양쪽 동의 시 7일 단위로 연장할 수 있다.
- MVP 기준 무료 사용자는 최대 14일, Plus 사용자는 최대 21일을 가설로 둔다.
- 대화방 만료 후에는 메시지를 보낼 수 없다.

---

## 5.10 연락처 교환

### 목적

사용자가 내부 대화 후 더 이어가고 싶을 때, 상호 동의 기반으로 연락처를 교환할 수 있게 한다.

### 정책

- 사용자는 내부 대화방에서 연락처 교환을 요청할 수 있다.
- 상대가 동의해야 연락처가 공개된다.
- 일방 요청만으로는 연락처가 공개되지 않는다.
- 연락처 교환 요청 전 공개될 정보와 범위를 명확히 안내한다.
- MVP에서는 전화번호 또는 카카오톡 ID 중 하나를 선택하는 구조를 검토할 수 있다.

---

## 5.11 신고/차단

### 신고

- 사용자는 프로필 상세와 대화방에서 신고할 수 있다.
- 신고 사유를 선택하고 필요 시 상세 내용을 입력할 수 있다.
- 신고 접수 후 사용자는 접수 완료 안내를 받는다.
- 운영자는 신고 내역과 관련 사용자/대화 데이터를 확인할 수 있어야 한다.

### 차단

- 사용자는 상대를 차단할 수 있다.
- 차단 즉시 상대와의 대화가 제한된다.
- 차단한 상대는 이후 추천 후보에서 제외된다.
- 차단 해제 기능은 MVP 후순위로 둘 수 있다.

# 8. MVP 성공 기준 가설

사전신청폼과 초기 지표 설계서는 별도 문서에서 다루되, MVP 기능 설계를 위해 최소한의 성공 기준 가설은 둔다.

| 영역 | 성공 신호 |
|---|---|
| 가입 | 랜딩 유입자가 사전 신청 또는 가입까지 이동 |
| 인증 | 본인/얼굴 인증 완료율이 충분히 높음 |
| 프로필 | 필수 프로필 작성 완료율이 충분히 높음 |
| 추천 | 추천 카드 열람률이 높음 |
| 리포트 | 매칭 리포트 열람률이 높음 |
| 관심 | 추천 대비 관심 보내기 비율이 발생 |
| 거절 | 거절 사유 수집률이 높고 이탈이 낮음 |
| 매칭 | 상호 관심 매칭이 발생 |
| 대화 | 매칭 후 첫 메시지 발송률이 높음 |
| 안전 | 신고/차단 접근성이 충분하고 운영 대응 가능 |
