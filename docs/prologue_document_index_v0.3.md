# 1. 문서 목적

이 문서는 프롤로그 프로젝트의 전체 기획 문서 목록, 문서 상태, 작성 우선순위, 문서 간 관계를 관리하기 위한 인덱스 문서다.

프롤로그는 서비스 전략, 브랜드, 제품 경험, 신뢰/안전, BM, 런칭, 개발 전환, 확장 전략 등 여러 문서로 확장된다. 문서가 늘어날수록 내용이 중복되거나 서로 충돌할 수 있으므로, 이 인덱스를 기준으로 전체 문서 구조를 관리한다.

# 2. 문서 관리 원칙

## 2.1 기준 문서와 하위 문서를 분리한다

**프롤로그 서비스 콘셉트 기획서 v1**은 최상위 기준 문서로 유지한다. 이 문서는 서비스의 정체성, 타깃, 포지셔닝, 브랜드 콘셉트, 핵심 가치, 핵심 경험을 정의한다.

세부 기능, 정책, 화면, BM, 랜딩페이지 등은 별도 하위 문서로 분리한다.

## 2.2 한 문서에는 하나의 목적만 둔다

문서가 비대해지는 것을 막기 위해 각 문서는 명확한 역할을 가진다.

예를 들어 인증과 관련된 내용은 여러 문서에 등장할 수 있지만, 각 문서의 역할은 다르다.

| 문서 | 인증 내용을 다루는 방식 |
|---|---|
| 서비스 콘셉트 기획서 | 인증이 왜 중요한지와 큰 방향만 설명 |
| 인증 정책 기획서 | 필수/선택/프리미엄 인증, 인증 배지, 실패 정책 상세 정의 |
| 온보딩 기획서 | 사용자가 언제 어떤 순서로 인증하는지 정의 |
| 개인정보/공개 범위 정책서 | 인증 데이터 수집, 보관, 삭제, 공개 범위 정의 |
| 랜딩페이지 기획서 | 사용자에게 인증을 어떻게 설명할지 카피화 |

## 2.3 중요한 결정은 의사결정 로그에 남긴다

서비스 방향, 정책, 네이밍, BM, 법적 포지션 등 주요 결정은 별도의 **의사결정 로그**에 기록한다.

결정 로그에는 다음 항목을 남긴다.

- 결정 날짜
- 결정 사항
- 결정 이유
- 영향받는 문서
- 추후 재검토 필요 여부

## 2.4 문서 상태를 명확히 표시한다

모든 문서는 상태값을 가진다.

| 상태 | 의미 |
|---|---|
| Draft | 초안. 계속 변경 가능 |
| Review | 방향은 잡혔지만 검토 중 |
| Confirmed | 현재 기준 확정 |
| Needs Update | 다른 결정으로 인해 업데이트 필요 |
| Deprecated | 더 이상 사용하지 않는 문서 |

# 16. 현재 별도 검토가 필요한 쟁점

다음 쟁점은 문서 작성 전에 추가 논의가 필요하다.

1. 현재 매칭 플로우의 최종 구조
2. 무료 사용자와 유료 사용자가 서로 매칭될 수 있는 구조인지 여부
3. 채팅을 서비스 내부에 구현할지 여부
4. 채팅을 외부 메신저로 전환해도 서비스 경험이 유지되는지 여부
5. 매칭 후 타로/사주 기반 궁합 부가 서비스를 제공할지 여부
6. 추가 추천권의 최종 가격, 월 구매 한도, 노출 방식
7. Plus와 Trust Plus의 실제 출시 시점과 구분 기준

# 17. 문서 업데이트 프로세스

앞으로 문서 수정은 다음 순서로 진행한다.

1. 대화에서 방향을 논의한다.
2. 결정된 내용을 짧게 요약한다.
3. 영향을 받는 문서를 확인한다.
4. 해당 문서를 업데이트한다.
5. 필요한 경우 문서 인덱스 상태를 업데이트한다.
6. 중요한 결정은 의사결정 로그에 기록한다.

---

# 18. 세션 변경 대비 운영 원칙

세션이나 대화가 달라질 수 있으므로, 문서 복구와 이어서 작업을 위해 다음 원칙을 둔다.

1. 중요한 문서는 항상 문서 인덱스에 등록한다.
2. 중요한 결정은 의사결정 로그에 남긴다.
3. 새 세션에서 이어갈 때는 서비스 콘셉트 기획서, 문서 인덱스, 의사결정 로그를 우선 참조한다.
4. 최종 보관은 추후 Notion, Google Docs, GitHub Markdown 등 외부 저장소로 이전할 수 있다.
5. 외부 저장소로 이전하기 전까지는 현재 ChatGPT 대화와 캔버스 문서를 1차 작업 공간으로 사용한다.

# 19. 다음 작업 제안

현재 기준으로 전략, 브랜드, 제품 경험, 신뢰안전, BM 초안은 1차 작성되었다.
