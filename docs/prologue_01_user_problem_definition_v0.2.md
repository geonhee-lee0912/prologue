# 1. 문서 목적

이 문서는 프롤로그가 해결해야 할 사용자 문제를 명확히 정의하기 위한 문서다.

프롤로그는 단순히 “소개팅 앱을 더 고급스럽게 만든 서비스”가 아니다. 기존 소개팅 앱의 접근성은 유지하되, 사용자가 반복적으로 경험하는 불신, 목적 불일치, 무한 탐색 피로, 사진/조건 중심 판단, 안전 불안, 매칭 후 대화 방치 문제를 줄이는 것을 목표로 한다.

또한 프롤로그는 결혼정보회사나 결정사를 그대로 앱으로 옮기는 서비스도 아니다. 결정사의 진지함과 신뢰 요소는 참고하되, 높은 비용, 무거운 가입 과정, 조건표 중심의 만남 방식은 그대로 가져오지 않는다.

이 문서는 다음 의사결정의 기준으로 활용한다.

1. 프롤로그가 반드시 해결해야 할 핵심 문제가 무엇인지 정한다.
2. 해결하지 않을 문제와 MVP에서 제외할 문제를 분리한다.
3. 기능 우선순위와 정책 우선순위를 판단한다.
4. 랜딩페이지, 온보딩, 추천 카드, 매칭 리포트, 대화 경험의 메시지 방향을 정한다.
5. 사용자 리서치와 사전 신청폼에서 검증해야 할 질문을 설계한다.
6. 서비스가 과도하게 결정사화되거나, 반대로 기존 소개팅 앱처럼 가벼워지는 것을 방지한다.

---

# 2. 문제 정의 한 줄 요약

**프롤로그가 해결하려는 핵심 문제는, 사용자가 앱처럼 쉽게 만남을 시작하고 싶어 하면서도 기존 소개팅 앱에서는 상대의 신뢰도와 관계 목적을 확인하기 어렵고, 결정사는 너무 비싸고 무겁게 느낀다는 점이다.**

v0.2에서는 여기에 한 가지 문제를 더 명확히 추가한다.

**상호 매칭 이후에도 대화가 방치되거나, 너무 이른 연락처 교환 압박이 생기거나, 대화 종료 권한이 불명확하면 사용자는 다시 피로감과 불안을 느낀다.**

프롤로그의 문제 정의는 다음 문장으로 요약할 수 있다.

> 소개팅 앱은 너무 가볍고, 결정사는 너무 무겁다.  
> 프롤로그는 그 사이에서 신뢰할 수 있는 사람을, 부담스럽지 않게, 이야기로 만나고, 안전하게 대화를 시작할 수 있게 한다.

---

# 3. 사용자 문제의 출발점

프롤로그의 사용자는 단순히 “연애하고 싶은 사람”이 아니다. 이들은 이미 기존 소개팅 앱, 지인 소개, 소셜 네트워크, 결정사 등의 선택지를 알고 있거나 경험해본 사람들이다.

그럼에도 프롤로그가 필요해지는 이유는 다음과 같다.

1. 소개팅 앱은 쉽지만 믿기 어렵다.
2. 결정사는 믿을 수 있을 것 같지만 시작하기 부담스럽다.
3. 지인 소개는 자연스럽지만 기회가 제한적이다.
4. 오프라인 만남은 우연성이 크고 효율이 낮다.
5. 진지한 관계를 원하지만, 처음부터 결혼이나 조건 중심으로만 접근하고 싶지는 않다.
6. 매칭 이후에도 대화가 흐지부지되거나, 연락처 교환 과정이 불편하게 느껴질 수 있다.

따라서 프롤로그는 다음과 같은 사용자를 위한 서비스다.

- 가볍게 넘기는 소개팅 앱에 피로감을 느끼는 사람
- 상대가 실제 싱글인지, 신원이 맞는지, 관계 목적이 비슷한지 알고 싶은 사람
- 결정사처럼 무거운 상담과 고가 계약은 부담스러운 사람
- 사진과 조건만이 아니라, 사람의 이야기와 관계 성향을 보고 싶은 사람
- 무한 스와이프보다 신중한 추천을 받고 싶은 사람
- 매칭 후에도 안전하고 부담 없는 대화 시작 구조를 원하는 사람

# 4. 핵심 사용자 문제 구조

프롤로그가 다루는 사용자 문제는 크게 여섯 가지로 나눌 수 있다.

| 문제 영역 | 사용자가 느끼는 문제 | 프롤로그의 대응 방향 |
|---|---|---|
| 신뢰 문제 | 프로필이 진짜인지 믿기 어렵다 | 본인·얼굴·관계 목적·직업/재직 인증 |
| 목적 문제 | 상대가 어떤 만남을 원하는지 알기 어렵다 | 관계 목적 설문, 싱글 상태 서약, 만남 태도 확인 |
| 피로 문제 | 무한 스와이프와 반복 대화가 피곤하다 | 제한된 큐레이션 추천, 추천 이유 제공 |
| 판단 문제 | 사진과 조건만으로 판단하게 된다 | 이야기형 프로필, 관계 성향, 라이프스타일 정보 |
| 안전 문제 | 비매너, 기혼자, 허위 정보, 정보 노출이 불안하다 | 신고/차단/제재, 정보 공개 범위, 안전 UX |
| 대화 전환 문제 | 매칭 후 대화가 방치되거나 연락처 교환이 부담스럽다 | 7일 기본 대화방, 상호 동의 연장, 상호 동의 연락처 교환 |

이 여섯 가지 문제는 서로 분리되어 있지 않다. 예를 들어 프로필을 신뢰할 수 없으면 추천 이유도 믿기 어렵고, 관계 목적이 불명확하면 대화가 쉽게 피로해진다. 또 상호 매칭이 되더라도 대화방이 방치되거나 연락처 교환 압박이 생기면 사용자는 다시 기존 소개팅 앱과 비슷한 피로를 느낄 수 있다.

# 5. 문제 1: 프로필과 신원에 대한 불신

## 5.1 사용자 문제

기존 소개팅 앱에서 사용자는 상대의 프로필을 온전히 신뢰하기 어렵다.

- 사진이 실제 본인 사진인지 알기 어렵다.
- 나이, 직업, 거주지 정보가 사실인지 확신하기 어렵다.
- 상대가 실제로 싱글인지 알 수 없다.
- 직업이나 생활 수준을 과장했을 수 있다.
- 프로필이 너무 적어 판단할 근거가 부족하다.
- 매칭 후 대화에서야 정보가 맞지 않다는 것을 알게 된다.

## 5.2 프롤로그의 해결 방향

1. 본인 인증과 얼굴 인증을 기본 조건으로 둔다.
2. 나이와 기본 신원은 최소 수준에서 확인한다.
3. 관계 목적 설문과 싱글 상태 서약을 통해 불성실한 사용자를 줄인다.
4. 직업/재직 인증은 강력 권장 인증으로 두고 추천 품질에 반영한다.
5. 소득이나 경제 안정성 인증은 초기 필수가 아니라 선택 또는 프리미엄 풀 조건으로 둔다.
6. 원본 정보가 아니라 인증 결과와 신뢰 배지를 중심으로 보여준다.

## 5.3 주의할 점

프롤로그는 “조건 검증 서비스”처럼 보여서는 안 된다. 인증은 사람을 등급화하거나 조건으로 줄 세우기 위한 장치가 아니라, 허위 프로필과 불성실한 만남을 줄이기 위한 최소 신뢰 장치로 설명해야 한다.

# 6. 문제 2: 관계 목적의 불명확성

## 6.1 사용자 문제

소개팅 앱에서는 같은 앱을 쓰는 사용자라도 원하는 관계의 온도와 목적이 크게 다를 수 있다. 어떤 사용자는 가볍게 대화하거나 즉흥적인 만남을 원하고, 어떤 사용자는 진지한 연애를 원한다. 또 어떤 사용자는 장기적인 관계 가능성까지 열어두고 있다.

문제는 이 차이가 매칭 전에는 잘 드러나지 않는다는 점이다.

## 6.2 프롤로그의 해결 방향

1. 가입 또는 추천 전 관계 목적 설문을 진행한다.
2. “결혼 상대”, “배우자”, “성혼” 같은 표현은 초기 전면에서 사용하지 않는다.
3. 사용자가 원하는 관계의 속도, 연락 방식, 장기 관계 가능성을 파악한다.
4. 관계 목적이 지나치게 다른 사용자는 추천 우선순위를 낮춘다.
5. 추천 리포트에서 잘 맞는 지점과 확인해야 할 차이점을 함께 보여준다.

---

# 7. 문제 3: 무한 스와이프와 반복 대화의 피로감

## 7.1 사용자 문제

기존 소개팅 앱은 많은 사람을 빠르게 보여주는 방식으로 작동한다. 이 구조는 접근성과 재미는 높이지만, 시간이 지날수록 피로를 만든다.

- 너무 많은 프로필을 빠르게 넘기게 된다.
- 한 사람을 깊게 보기보다 사진만 보고 판단하게 된다.
- 매칭은 되지만 대화가 이어지지 않는다.
- 반복적으로 비슷한 자기소개와 대화를 하게 된다.
- 대화가 실제 만남으로 연결되지 않는다.
- 앱을 오래 쓸수록 사람을 소비하는 느낌이 든다.

## 7.2 프롤로그의 해결 방향

1. 하루 또는 일정 기간 동안 제공되는 추천 수를 제한한다.
2. 추천 카드에 단순 프로필이 아니라 추천 이유를 함께 제공한다.
3. 기본 추천 소진 후에는 추가 추천권으로 큐레이션 후보를 추가 열람할 수 있게 한다.
4. 추가 추천권은 무한 탐색권이 아니라 제한된 큐레이션 추천을 추가로 확인하는 상품으로 운영한다.
5. 사용자가 상대를 넘길 때 거절 사유를 수집해 추천 품질을 개선한다.
6. 관심 표현을 가볍게 남발하는 구조보다 신중한 선택을 유도한다.

## 7.3 주의할 점

추천 수를 제한하면 사용자는 추천 품질에 더 민감해진다. 따라서 제한된 추천과 추가 추천권은 반드시 납득 가능한 추천 이유, 충분한 프로필 정보, 개선 가능한 피드백 구조와 함께 설계되어야 한다.

# 8. 문제 4: 사진과 조건 중심 판단의 한계

## 8.1 사용자 문제

기존 소개팅 앱은 사진, 나이, 지역, 직업 등 짧은 정보만으로 상대를 판단하게 만든다. 이 구조에서는 빠른 선택은 가능하지만, 사람을 깊이 이해하기 어렵다.

## 8.2 프롤로그의 해결 방향

1. 자기소개를 “나의 프롤로그”로 구성한다.
2. 관계 성향, 라이프스타일, 대화 스타일, 가치관을 프로필에 포함한다.
3. 추천 카드에서 조건 정보와 이야기 정보를 함께 보여준다.
4. 매칭 리포트는 점수표가 아니라 추천 이유와 관계 가능성을 설명한다.
5. 책방/서점 메타포를 통해 한 사람의 이야기를 읽는 경험을 만든다.

## 8.3 주의할 점

이야기형 프로필은 차별점이지만 작성 부담을 만들 수 있다. 따라서 처음부터 긴 글을 요구하기보다, 짧은 문장형 문답과 선택형 질문을 섞어 작성 부담을 줄여야 한다.

---

# 9. 문제 5: 안전감과 통제감 부족

## 9.1 사용자 문제

소개팅 서비스에서 안전감은 단순한 부가 기능이 아니라 핵심 신뢰 요소다. 특히 여성 사용자는 상대의 신원, 대화 태도, 정보 노출, 신고 대응에 더 민감할 수 있다.

- 내 사진이 캡처되거나 외부에 공유될까 걱정된다.
- 상대가 무례하게 대화하거나 집요하게 연락할 수 있다.
- 신고해도 제대로 처리될지 모르겠다.
- 차단 이후에도 정보가 남아 있을까 불안하다.
- 기혼자나 연인 있는 사용자가 섞여 있을 수 있다.
- 내가 원하지 않는 정보까지 공개될까 걱정된다.

## 9.2 프롤로그의 해결 방향

1. 정보 공개 범위를 단계적으로 설정할 수 있게 한다.
2. 신고, 차단, 제재 정책을 명확하게 안내한다.
3. 싱글 상태 서약과 신고 기반 제재를 결합한다.
4. 비매너, 허위 정보, 노쇼에 대한 운영 기준을 만든다.
5. 안전 관련 화면에서는 캐릭터나 감성 표현보다 명확한 문구를 우선한다.

# 10. 문제 6: 매칭 후 대화 방치와 연락처 전환 불안

## 10.1 사용자 문제

상호 관심이 발생해도 좋은 만남으로 이어지는 것은 아니다. 사용자는 매칭 후에도 다음 문제를 경험할 수 있다.

- 매칭만 되고 첫 메시지가 오가지 않는다.
- 한쪽만 계속 대화를 이어가려 하고 다른 한쪽은 부담을 느낀다.
- 너무 이른 연락처 교환이나 외부 메신저 이동을 요구받는다.
- 대화방이 무기한 남아 있어 불편하다.
- 대화 종료나 차단을 어떻게 해야 할지 명확하지 않다.
- 대화가 끝난 뒤에도 상대에게 내 정보가 계속 보일까 걱정된다.

## 10.2 사용자 감정

- “매칭은 됐는데 또 흐지부지되는 건 아닐까?”
- “상대가 연락처를 너무 빨리 요구하면 부담스럽다.”
- “대화를 더 이어갈지, 여기서 멈출지 내가 선택할 수 있으면 좋겠다.”
- “대화방이 언제까지 열려 있는지 알고 싶다.”

## 10.3 프롤로그의 해결 방향

프롤로그는 매칭 후 내부 대화방을 무한정 열어두지 않는다. 대화방은 서로를 알아가기 위한 임시 공간이며, 일정 기간 안에 대화를 이어갈지, 연락처를 교환할지, 자연스럽게 종료할지 선택할 수 있어야 한다.

권장 정책은 다음과 같다.

| 항목 | 정책 |
|---|---|
| 채팅 오픈 조건 | 상호 관심 발생 |
| 기본 대화 기간 | 7일 |
| 연장 방식 | 양쪽 동의 시 7일 단위 연장 |
| MVP 최대 기간 | 무료 14일, Plus 21일 |
| 정식 서비스 검토안 | 무료 21일, Plus 30일 |
| 연락처 교환 | 상호 동의형 |
| 메시지 수 제한 | 두지 않음 |
| 신고/차단 | 항상 제공 |
| 대화 종료 | 언제든 가능 |

## 10.4 제품 반영

- 매칭 직후 “첫문장이 이어졌어요” 화면 제공
- 대화 전 매너 안내 제공
- 기본 대화 기간과 남은 기간 표시
- 종료 전 연장 또는 연락처 교환 선택지 제공
- 연락처 교환은 한쪽 요청, 상대 수락 후에만 공개
- 대화 종료 후 짧은 피드백 수집
- 신고/차단 버튼을 대화방 내 상시 제공

## 10.5 주의할 점

대화방 기간 제한은 사용자를 압박하는 장치처럼 보이면 안 된다. “놓치면 끝”이라는 식의 FOMO 카피가 아니라, 서로의 속도를 존중하고 방치된 대화를 줄이기 위한 안전한 운영 정책으로 설명해야 한다.

# 11. 세그먼트별 주요 문제

## 11.1 26~30세: 진지한 연애 시작층

- 소개팅 앱은 가볍지만 결정사는 너무 이르다.
- 결혼을 강요받고 싶지는 않지만 가벼운 만남도 싫다.
- 인증이 너무 무겁거나 조건 중심이면 부담스럽다.
- 대화방 제한은 압박이 아니라 자연스러운 속도 조절로 느껴져야 한다.

필요한 해결은 부담스럽지 않은 진지함, 관계 목적 확인, 쉬운 온보딩, 이야기형 프로필, 첫 대화 주제 제공이다.

## 11.2 30~35세: 관계 안정 추구층

- 더 이상 의미 없는 매칭과 대화에 시간을 쓰고 싶지 않다.
- 신뢰 가능한 프로필과 관계 목적 확인이 필요하다.
- 상대가 나와 실제로 맞을 가능성이 있는지 알고 싶다.
- 추가 추천권은 무한 탐색이 아니라 더 납득되는 추천 기회로 제공되어야 한다.
- 대화방 기간 제한과 상호 동의 연락처 교환은 시간 낭비와 불편함을 줄이는 장치가 될 수 있다.

필요한 해결은 인증, 제한된 큐레이션 추천, 상세 매칭 리포트, 첫 대화 보조, 신중한 연락처 전환이다.

## 11.3 35~39세: 관계 명확성 추구층

- 목적이 불분명한 사람에게 시간을 쓰고 싶지 않다.
- 상대의 기본 정보와 관계 의향이 신뢰 가능해야 한다.
- 추천이 적어도 품질이 높아야 한다.
- 대화가 장기간 방치되기보다 명확한 선택지와 종료 조건이 있는 편이 더 효율적으로 느껴질 수 있다.

필요한 해결은 인증 배지, 관계 목적 확인, 고신뢰 풀, 실질적인 추천 근거, 명확한 대화 종료/연장 정책이다.

# 12. 성별 관점의 문제 정의

## 12.1 여성 사용자의 문제

여성 사용자는 소개팅 서비스에서 상대의 신뢰도, 정보 노출, 대화 안전, 신고 대응, 연락처 교환 통제에 특히 민감할 수 있다.

추가로 중요한 문제는 다음과 같다.

- 상호 매칭 후 상대가 너무 빠르게 연락처를 요구할 수 있다.
- 대화방이 무기한 열려 있으면 부담스럽다.
- 대화를 끝내고 싶어도 상대에게 나쁜 인상을 줄까 걱정된다.
- 외부 연락처 전환 후에는 서비스가 보호해주기 어렵다고 느낄 수 있다.

프롤로그는 여성 사용자가 “상호 매칭 이후에도 내가 통제할 수 있다”고 느끼게 해야 한다.

## 12.2 남성 사용자의 문제

남성 사용자는 소개팅 앱에서 과도한 경쟁, 낮은 매칭 효율, 의미 없는 좋아요 소비에 피로감을 느낄 수 있다.

추가로 중요한 문제는 다음과 같다.

- 매칭 후 첫 메시지를 어떻게 시작해야 할지 어렵다.
- 상대가 답장을 하지 않을 때 언제까지 기다려야 할지 불분명하다.
- 유료 결제의 가치가 단순 노출이나 좋아요 구매로만 느껴지면 납득하기 어렵다.

프롤로그는 남성 사용자에게 “인증과 프로필 완성도, 추가 추천권, 첫 대화 보조 기능이 실제 대화 기회를 높인다”고 느끼게 해야 한다.

# 13. 프롤로그가 해결하지 않을 문제

프롤로그는 모든 연애 문제를 해결하는 서비스가 아니다. 초기 MVP에서 해결하지 않을 문제를 명확히 해야 제품 범위가 흔들리지 않는다.

1. 성혼 보장
2. 전담 매니저식 수동 소개
3. 무제한 탐색 욕구
4. 가벼운 만남과 즉흥 만남
5. 완벽한 신원 보증
6. 커뮤니티와 소셜 피드
7. 사주/타로를 핵심 매칭 근거로 사용하는 것
8. 무기한 대화방 유지

타로/사주 궁합은 향후 매칭 후 대화 보조용 재미 콘텐츠로 검토할 수 있으나, 추천 알고리즘의 핵심 근거가 되어서는 안 된다.

# 15. MVP에서 우선 해결할 문제

## 15.1 1순위 문제

1. 상대가 실제 사람이고 기본 신뢰가 있는지 확인하기 어렵다.
2. 상대가 어떤 관계를 원하는지 알기 어렵다.
3. 사진과 조건만으로 판단하게 된다.
4. 무한 스와이프와 의미 없는 매칭이 피곤하다.
5. 왜 이 사람이 추천됐는지 알 수 없다.
6. 허위 정보, 비매너, 기혼자 리스크가 불안하다.

## 15.2 2순위 문제

1. 사진 공개 범위를 세밀하게 통제하고 싶다.
2. 대화가 자연스럽게 시작되도록 도움을 받고 싶다.
3. 대화방이 방치되지 않고 자연스럽게 종료/연장되기를 원한다.
4. 연락처 교환이 상호 동의로만 이루어지기를 원한다.
5. 추천 품질이 내 피드백에 따라 개선되길 원한다.
6. 더 신뢰도 높은 사람만 모인 풀을 이용하고 싶다.

## 15.3 MVP 이후 문제

1. 고신뢰 풀 또는 프리미엄 풀 설계
2. 상세 매칭 리포트 고도화
3. 프로필 개선 리포트
4. 타로/사주 기반 대화 보조 콘텐츠
5. 국제 관계 매칭 확장
6. 오프라인 이벤트 또는 커뮤니티 기능

# 16. 사용자 문제 검증 질문

사전 신청폼, 사용자 인터뷰, 베타 테스트에서 다음 질문을 통해 문제의 강도와 우선순위를 검증한다.

## 16.1 소개팅 앱 경험 질문

1. 기존 소개팅 앱을 사용해본 적이 있나요?
2. 사용해봤다면 가장 불편했던 점은 무엇인가요?
3. 매칭 후 실제 대화나 만남으로 이어진 비율은 어느 정도였나요?
4. 상대의 프로필을 얼마나 신뢰할 수 있다고 느꼈나요?
5. 사진 중심의 탐색 방식에 피로감을 느낀 적이 있나요?

## 16.2 신뢰와 인증 관련 질문

1. 본인 인증과 얼굴 인증이 필수라면 가입할 의향이 있나요?
2. 직업/재직 인증이 추천 품질에 영향을 준다면 인증할 의향이 있나요?
3. 싱글 상태 서약 또는 확인 절차가 필요하다고 느끼나요?
4. 소득 또는 경제 안정성 인증은 어떤 상황에서 수용할 수 있나요?
5. 인증 배지가 있는 상대에게 더 관심을 보낼 가능성이 있나요?

## 16.3 큐레이션과 과금 관련 질문

1. 하루 추천 수가 제한되어도 추천 이유가 충분하다면 사용할 의향이 있나요?
2. 기본 추천을 모두 본 뒤 추가 추천권을 구매할 의향이 있나요?
3. 추가 추천권이 “무한 탐색”이 아니라 “큐레이션 추천 추가 열람”이라면 납득되나요?
4. 상세 매칭 리포트가 유료 가치로 느껴지나요?
5. 무료 사용자와 유료 사용자가 서로 매칭되는 구조가 괜찮다고 느끼나요?

## 16.4 대화 경험 관련 질문

1. 상호 매칭 후 7일 기본 대화방이 적절하다고 느끼나요?
2. 양쪽 동의가 있을 때만 대화방이 연장되는 구조가 안전하게 느껴지나요?
3. 연락처 교환이 상호 동의형이어야 한다고 느끼나요?
4. 사진/영상/위치 공유를 초기에는 제한하는 것에 동의하나요?
5. 매칭 후 타로/사주 궁합 카드가 대화 시작에 도움이 될 것 같나요?

# 17. 문제 해결 성공 기준

## 17.1 정량 지표

- 사전 신청 전환율
- 본인/얼굴 인증 완료율
- 관계 목적 설문 완료율
- 프로필 작성 완료율
- 추천 카드 열람률
- 매칭 리포트 열람률
- 관심 보내기 비율
- 상호 관심 전환율
- 기본 대화 시작률
- 첫 24시간 응답률
- 7일 대화 지속률
- 대화 연장 요청률
- 연락처 교환 상호 동의율
- 대화 후 피드백 제출률
- 신고/차단 발생률
- 추가 추천권 구매율

## 17.2 정성 지표

- “기존 앱보다 신뢰된다”는 응답
- “상대의 관계 목적을 알 수 있어 좋다”는 응답
- “추천 이유가 납득된다”는 응답
- “사진만 보고 판단하는 느낌이 덜하다”는 응답
- “내 정보가 안전하게 관리된다고 느낀다”는 응답
- “대화방 종료/연장 조건이 명확해서 좋다”는 응답
- “연락처 교환을 내가 통제할 수 있어 안심된다”는 응답

## 17.3 실패 신호

- 인증 과정에서 이탈률이 지나치게 높다.
- 사용자가 프롤로그를 결정사처럼 너무 무겁게 인식한다.
- 추천 수가 적다는 불만이 추천 품질 만족도보다 크다.
- 매칭 리포트를 읽지 않고 바로 넘기는 비율이 높다.
- 내부 대화방 기간 제한이 압박으로 받아들여진다.
- 연락처 교환 요청이 불편하거나 안전하지 않게 느껴진다.
- 타로/사주 궁합이 핵심 추천 근거로 오해된다.

1. 본인 인증과 얼굴 인증의 구체적 방식
2. 직업/재직 인증의 필수 여부와 인증 실패 정책
3. 싱글 상태 서약의 문구와 위반 시 제재 기준
4. 사진 공개 범위와 캡처 방지 정책
5. 추천 수 제한의 구체적 기준
6. 추가 추천권의 최종 가격과 월 구매 한도
7. 매칭 리포트의 정보 깊이와 노출 범위
8. 거절 사유 선택지의 최종 항목
9. 대화방 기본 기간과 최대 기간의 최종 정책
10. 연락처 교환 시 공개 정보 범위
11. 타로/사주 궁합 카드의 도입 여부와 표현 방식
12. 유료화 시 사용자 문제가 실제 지불 가치로 전환되는 지점
