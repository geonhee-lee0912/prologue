# 1. 문서 목적

이 문서는 프롤로그의 인증 정책을 정의하기 위한 문서다.

프롤로그의 인증은 결혼 조건 검증이나 사용자 등급화를 위한 장치가 아니다. 인증은 허위 프로필, 도용 사진, 목적 불명확성, 기혼자/연인 있음 리스크, 비매너 사용자를 줄이기 위한 신뢰 장치다.

인증 정책은 서비스의 핵심 포지셔닝과 직접 연결된다. 인증이 너무 약하면 일반 소개팅 앱과 다르지 않게 느껴지고, 인증이 너무 무겁거나 조건 중심이면 결정사처럼 보일 수 있다. 따라서 프롤로그는 “필수 인증은 명확하게, 민감 인증은 단계적으로, 공개는 부드럽게” 설계한다.

이 문서는 다음 의사결정의 기준으로 활용한다.

1. 필수/권장/선택 인증을 구분한다.
2. 인증 배지와 프로필 표시 방식을 정의한다.
3. 인증 실패, 재시도, 만료, 허위 인증 대응 정책을 정한다.
4. 온보딩과 추천 품질에서 인증을 어떻게 활용할지 정의한다.
5. 개인정보와 공개 범위 원칙의 기준을 만든다.
6. 향후 고신뢰 풀 또는 프리미엄 BM과 연결할 인증 후보를 정리한다.

---

# 2. 인증 정책 한 줄 정의

**프롤로그의 인증은 사람을 조건으로 평가하기 위한 것이 아니라, 사용자가 덜 의심하고 진지한 만남을 시작할 수 있도록 돕는 신뢰 장치다.**

핵심 원칙은 다음과 같다.

> 인증은 강하게, 공개는 부드럽게.

---

# 3. 인증 설계 원칙

## 3.1 인증은 신뢰의 출발선이다

프롤로그에서 인증은 상대를 완벽히 보증하는 것이 아니라, 만남을 시작하기 위한 최소 신뢰 출발선을 만드는 역할을 한다.

## 3.2 인증 원본보다 결과를 보여준다

사용자에게 민감한 원본 정보가 그대로 노출되어서는 안 된다. 상대에게는 인증 완료 여부, 배지, 요약 상태만 보여준다.

## 3.3 필수와 선택을 분리한다

모든 인증을 필수로 두면 가입 장벽이 높아진다. 허위 계정과 도용 사진 방지에 필요한 최소 인증은 필수로 두고, 민감하거나 부담이 큰 인증은 선택 또는 프리미엄 풀 조건으로 둔다.

## 3.4 인증은 추천 품질과 연결한다

인증 완료자는 더 신뢰도 높은 추천 경험을 얻을 수 있어야 한다. 다만 인증을 많이 했다고 사람의 가치가 높다는 식으로 표현해서는 안 된다.

## 3.5 인증은 법적/개인정보 리스크를 고려한다

소득, 학력, 직장, 싱글 상태 증빙 등은 민감하거나 운영 리스크가 있다. 수집 목적, 보관 기간, 공개 범위, 삭제 정책을 명확히 해야 한다.

# 4. 인증 분류 체계

프롤로그의 인증은 세 단계로 나눈다.

| 분류 | 의미 | 예시 | MVP 우선순위 |
|---|---|---|---:|
| 필수 인증 | 추천을 받기 위해 반드시 필요한 인증 | 본인, 얼굴, 나이, 관계 목적, 싱글 서약, 매너 서약 | 1 |
| 강력 권장 인증 | 추천 품질과 신뢰도에 영향을 주는 인증 | 직업/재직, 거주 지역, 학력 | 2 |
| 선택/프리미엄 인증 | 민감도가 높거나 고신뢰 풀에 활용할 인증 | 소득/경제 안정성, 싱글 상태 증빙 | 3 |

# 5. 필수 인증

## 5.1 본인 인증

### 목적

가짜 계정, 중복 계정, 악성 계정을 줄인다.

### 방식 후보

- 휴대폰 본인 인증
- 통신사 인증
- 외부 본인 인증 솔루션

### 표시 방식

- 본인 인증 완료 배지
- 실명은 공개하지 않음

### 정책

- 추천을 받기 위한 필수 조건으로 둔다.
- 본인 인증 미완료 사용자는 추천 카드 노출 대상에서 제외한다.
- 중복 계정 탐지 기준과 제재 정책은 별도 정의가 필요하다.

### 사용자 안내 문구

> 본인 인증은 허위 계정을 줄이고, 더 신뢰할 수 있는 추천을 만들기 위한 과정입니다.

---

## 5.2 얼굴 인증

### 목적

도용 사진, 타인 사칭, 실물과 다른 프로필 사용을 줄인다.

### 방식 후보

- 실시간 셀피 촬영
- 프로필 사진과 셀피 비교
- 자동 비교 API
- 수동 검수

### 표시 방식

- 얼굴 인증 완료 배지
- 인증용 셀피는 상대에게 공개하지 않음

### 정책

- 추천을 받기 위한 필수 조건으로 둔다.
- 얼굴 인증 실패 시 재촬영 가이드를 제공한다.
- 반복 실패 시 수동 검토 또는 고객지원 흐름을 제공한다.

### 사용자 안내 문구

> 얼굴 인증은 도용 사진을 줄이기 위한 과정이에요. 인증용 사진은 상대에게 공개되지 않습니다.

---

## 5.3 나이 인증

### 목적

서비스 이용 가능 연령과 타깃 연령을 관리한다.

### 방식

- 본인 인증 정보 기반 나이 확인
- 생년월일 직접 입력 후 본인 인증과 대조

### 표시 방식

- 나이만 표시
- 생년월일 원본은 공개하지 않음

### 정책

- 만 19세 미만은 이용 제한한다.
- 프롤로그의 기본 타깃은 26~39세이나, 실제 가입 허용 범위는 별도 정책으로 결정한다.
- 26~39세 외 사용자의 노출/추천 정책은 추후 검토한다.

---

## 5.4 관계 목적 확인

### 목적

가벼운 목적, 목적 불명확성, 관계 기대 불일치를 줄인다.

### 방식

- 관계 목적 설문
- 관계 시작 속도 질문
- 연락 빈도 질문
- 장기 관계 가능성 질문

### 표시 방식

- “관계 목적 확인 완료” 배지
- 프로필에는 요약 정보만 표시
- 상세 답변은 추천 알고리즘과 매칭 리포트에 활용

### 정책

- 추천을 받기 위한 필수 조건으로 둔다.
- 결혼을 전면화하지 않고 관계 목적과 속도 중심으로 묻는다.

### 사용자 안내 문구

> 관계를 시작하는 속도와 온도가 비슷한 사람을 추천하기 위한 질문이에요.

---

## 5.5 싱글 상태 서약

### 목적

기혼자, 연인 있음, 불성실한 관계 상태의 사용자를 줄인다.

### 방식

- 체크박스 서약
- 서약 문구 동의
- 위반 시 제재 안내

### 표시 방식

- 싱글 상태 서약 완료 배지
- 법적 혼인상태를 직접 표시하지 않음

### 서약 문구 초안

> 현재 배우자 또는 연인이 없으며, 새로운 관계를 시작할 의사가 있음을 확인합니다.

### 정책

- 초기에는 증빙 인증보다 서약 중심으로 운영한다.
- 허위 서약 신고가 접수되면 검토 후 제재할 수 있다.
- 고신뢰 풀에서는 증빙 인증 도입을 검토한다.

---

## 5.6 매너 서약

### 목적

비매너 대화, 잠수, 노쇼, 허위 정보 입력을 줄인다.

### 방식

- 온보딩 중 서약 동의
- 대화 시작 전 짧은 리마인드

### 표시 방식

- 매너 서약 완료 상태
- 프로필에 배지로 노출할지 여부는 추후 검토

### 서약 항목

- 허위 정보를 입력하지 않음
- 상대를 존중하는 태도로 대화함
- 불쾌하거나 위협적인 표현을 사용하지 않음
- 만남 약속을 가볍게 취소하거나 노쇼하지 않음
- 관계 상태를 속이지 않음

---

# 6. 강력 권장 인증

## 6.1 직업/재직 인증

### 목적

허위 직업 정보와 과장된 프로필을 줄인다.

### 방식 후보

- 재직증명서
- 명함/사원증 일부 마스킹
- 회사 이메일 인증
- 건강보험/4대보험 기반 확인: 법적/개인정보 검토 필요
- 외부 인증 솔루션

### 표시 방식

- 직업/재직 인증 완료 배지
- 회사명 공개는 선택
- 직업군은 기본 표시 가능

### 정책

- 초기에는 필수가 아니라 강력 권장 인증으로 둔다.
- 인증 완료자는 추천 신뢰도 또는 노출 우선순위에 긍정적으로 반영한다.
- 허위 직업 신고가 많아지면 필수 전환을 검토한다.

### 사용자 안내 문구

> 직업/재직 인증을 완료하면 프로필 신뢰도가 높아지고, 더 잘 맞는 추천에 활용될 수 있어요.

---

## 6.2 거주 지역 인증

### 목적

실제 생활권과 만남 가능성을 높인다.

### 방식 후보

- 휴대폰 위치 기반 일회성 확인
- 주소지 인증: 부담이 커서 MVP 우선순위 낮음
- 자주 활동하는 생활권 직접 입력

### 표시 방식

- 생활권 요약 표시
- 상세 주소 비공개

### 정책

- MVP에서는 직접 입력 중심으로 시작할 수 있다.
- 위치 기반 인증은 추후 실제 만남 가능성 문제가 커질 때 검토한다.

---

## 6.3 학력 인증

### 목적

일부 사용자에게 선택적 신뢰 정보를 제공한다.

### 방식 후보

- 학교 이메일 인증
- 졸업증명서 일부 마스킹
- 외부 인증 솔루션

### 정책

- MVP 필수로 두지 않는다.
- 조건 중심, 스펙 중심 이미지가 강해질 수 있으므로 노출 방식을 신중히 설계한다.
- 사용자가 공개 여부를 선택할 수 있어야 한다.

---

# 7. 선택/프리미엄 인증

## 7.1 소득/경제 안정성 인증

### 목적

고신뢰 풀 또는 프리미엄 매칭에서 경제 안정성에 대한 신뢰 수요를 충족한다.

### 정책 방향

- 초기 전체 사용자 필수 인증으로 두지 않는다.
- 선택 배지 또는 프리미엄 풀 조건으로 검토한다.
- 정확한 금액보다 구간 또는 인증 여부 중심으로 설계한다.
- 개인정보/법무 검토가 반드시 필요하다.

### 표시 방식 후보

- 경제 안정성 인증 완료
- 소득 구간 공개 여부 선택
- 정확한 금액 비공개

### 주의점

- 사람을 소득으로 등급화하는 인상을 주면 안 된다.
- 결정사처럼 보이는 리스크가 크다.
- 여성/남성 모두에게 불쾌하거나 부담스러울 수 있다.

---

## 7.2 싱글 상태 증빙

### 목적

고신뢰 풀에서 기혼자 또는 연인 있음 리스크를 더 낮춘다.

### 방식 후보

- 혼인관계증명서 등 공적 서류: 법무/개인정보 검토 필요
- 이혼/미혼 여부 직접 증빙: 매우 민감
- 서약 + 신고/제재 강화

### 정책 방향

- 초기 MVP에서는 증빙 인증을 필수로 두지 않는다.
- 고신뢰 풀 또는 특정 프리미엄 기능에서 검토한다.
- 표현은 “혼인상태 인증”보다 “싱글 상태 확인”을 우선 사용한다.

# 8. 인증 상태값

인증 항목은 다음 상태를 가진다.

| 상태 | 의미 |
|---|---|
| Not Started | 인증 시작 전 |
| Pending | 인증 제출 후 검토 중 |
| Verified | 인증 완료 |
| Failed | 인증 실패 |
| Retry Required | 재시도 필요 |
| Expired | 인증 유효 기간 만료 |
| Revoked | 허위 또는 문제로 인증 취소 |
| Hidden | 인증은 완료했지만 사용자 공개 비활성 |

# 9. 인증 실패/재시도 정책

## 9.1 실패 유형

| 인증 | 실패 원인 | 대응 |
|---|---|---|
| 본인 인증 | 정보 불일치, 인증 오류 | 재시도, 고객지원 |
| 얼굴 인증 | 얼굴 인식 실패, 사진 불일치 | 재촬영, 수동 검토 |
| 직업 인증 | 서류 불명확, 정보 불일치 | 보완 요청 |
| 학력 인증 | 인증 자료 부족 | 보완 요청 |
| 소득 인증 | 자료 불명확, 범위 초과 정보 | 보완 요청 또는 반려 |

## 9.2 재시도 원칙

- 실패 사유를 명확히 안내한다.
- 사용자를 비난하지 않는다.
- 재시도 방법을 구체적으로 제공한다.
- 반복 실패 시 고객지원 또는 수동 검토를 제공한다.

### 예시 문구

> 사진이 어두워 얼굴을 확인하기 어려웠어요. 밝은 곳에서 다시 촬영해주세요.

## 9.3 허위 인증 대응

허위 인증이 의심되거나 확인된 경우 다음 조치를 적용할 수 있다.

- 인증 취소
- 프로필 노출 제한
- 추천 제한
- 계정 일시 정지
- 영구 이용 제한
- 신고자 보호 조치

세부 제재 기준은 신고/차단/제재 정책서에서 정의한다.

# 10. 인증 배지 표시 정책

## 10.1 배지 표시 원칙

1. 핵심 배지는 추천 카드에 표시한다.
2. 상세 배지는 프로필 상세에서 확인할 수 있게 한다.
3. 배지는 사람을 등급화하지 않는다.
4. 배지 개수를 경쟁 요소로 만들지 않는다.
5. 민감 배지는 사용자가 공개 여부를 선택할 수 있어야 한다.

## 10.2 추천 카드 표시 배지

- 본인 인증
- 얼굴 인증
- 관계 목적 확인
- 싱글 상태 서약
- 직업/재직 인증: 완료 시 표시

## 10.3 상세 프로필 표시 배지

- 본인 인증 완료
- 얼굴 인증 완료
- 나이 확인 완료
- 관계 목적 확인 완료
- 싱글 상태 서약 완료
- 직업/재직 인증 완료
- 학력 인증 완료: 선택
- 경제 안정성 인증 완료: 선택/프리미엄

## 10.4 배지 문구 예시

권장:

- 본인 인증 완료
- 얼굴 인증 완료
- 관계 목적 확인 완료
- 직업/재직 인증 완료
- 싱글 상태 서약 완료

비권장:

- 상위 회원
- 고급 회원
- 스펙 검증 완료
- 결혼 조건 검증 완료
- 프리미엄 인재

# 11. 인증과 추천의 연결

인증은 추천 자격과 추천 품질에 영향을 줄 수 있다.

## 11.1 추천 자격

MVP에서는 다음 조건을 충족해야 추천을 받을 수 있다.

1. 본인 인증 완료
2. 얼굴 인증 완료
3. 나이 확인 완료
4. 관계 목적 설문 완료
5. 싱글 상태 서약 완료
6. 매너 서약 완료
7. 필수 프로필 작성 완료

## 11.2 추천 우선순위 후보

- 필수 인증 완료자 우선
- 직업/재직 인증 완료자 가점
- 프로필 완성도 높은 사용자 가점
- 신고/제재 이력 없는 사용자 가점
- 관계 목적이 명확한 사용자 가점

### 주의점

사용자에게 “점수”를 그대로 보여주지 않는다. 추천 품질 향상을 위한 내부 기준으로 활용한다.

# 12. 개인정보와 보관 원칙

이 문서에서는 인증 정책 차원의 기본 원칙만 정의하고, 상세 내용은 개인정보/공개범위 정책서에서 별도 정의한다.

## 12.1 기본 원칙

1. 인증 목적에 필요한 최소 정보만 수집한다.
2. 원본 정보 보관을 최소화한다.
3. 가능하면 검증 후 원본 삭제 또는 암호화 보관을 검토한다.
4. 상대에게는 인증 원본이 아닌 인증 상태만 보여준다.
5. 탈퇴 시 인증 데이터 삭제 정책을 명확히 한다.
6. 민감 정보는 별도 동의와 공개 범위 설정이 필요하다.

## 12.2 공개 원칙

| 정보 | 공개 방식 |
|---|---|
| 실명 | 비공개 또는 표시명 사용 |
| 생년월일 | 나이만 공개 |
| 본인 인증 원본 | 비공개 |
| 얼굴 인증 원본 | 비공개 |
| 직장 인증 원본 | 비공개 |
| 소득 인증 원본 | 비공개 |
| 인증 완료 여부 | 배지로 공개 |

# 13. 온보딩 내 인증 UX

## 13.1 인증 순서

MVP 추천 순서는 다음과 같다.

1. 본인 인증
2. 얼굴 인증
3. 싱글 상태 서약
4. 매너 서약
5. 관계 목적 확인
6. 프로필 작성
7. 선택 인증 안내

## 13.2 순서 설계 이유

- 본인/얼굴 인증으로 기본 신뢰를 먼저 확보한다.
- 싱글/매너 서약으로 서비스 문화와 기대 행동을 설정한다.
- 관계 목적 확인으로 추천 품질을 높인다.
- 선택 인증은 초기 이탈을 줄이기 위해 뒤로 배치한다.

## 13.3 인증 완료 후 보상

- 추천 준비 완료
- 신뢰 배지 표시
- 프로필 완성도 상승
- 선택 인증 안내
- 추천 품질 향상 기대 문구

# 15. 인증 리스크와 대응

## 15.1 가입 이탈 리스크

### 원인

- 인증 절차가 많음
- 개인정보 불안
- 결정사처럼 느껴짐

### 대응

- 필수 인증 수를 최소화한다.
- 각 인증의 목적을 명확히 설명한다.
- 선택 인증은 온보딩 이후 유도한다.

## 15.2 개인정보 리스크

### 원인

- 민감 정보 수집
- 원본 보관
- 공개 범위 불명확

### 대응

- 원본 최소 보관
- 배지 중심 표시
- 공개 범위 설정
- 탈퇴/삭제 정책 명확화

## 15.3 조건 중심 이미지 리스크

### 원인

- 직업, 학력, 소득 인증 강조
- 배지 개수 경쟁

### 대응

- 인증은 신뢰 장치로 설명한다.
- 이야기형 프로필과 관계 성향을 함께 강조한다.
- 민감 인증은 선택 또는 프리미엄 풀로 분리한다.

## 15.4 허위 인증 리스크

### 원인

- 조작 서류
- 대리 인증
- 사진 도용

### 대응

- 인증 방식 고도화
- 신고 기반 검토
- 인증 취소/제재
- 반복 문제 계정 제한
